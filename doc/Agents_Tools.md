## 1) 기업 서비스 구조화 에이전트

### A. 신청서 템플릿/구조 파서 tool

- 입력: 신청서 파일(PDF/DOCX/HWP) 또는 템플릿 ID
- 출력: 섹션/문항/필드 구조(계층), 필드 타입(텍스트/표/체크/첨부), 필수 여부

### B. UI Form Schema 생성 tool

- 입력: A의 필드 구조
- 출력: 프론트에서 바로 렌더 가능한 **Form JSON Schema**
  - 필드 ID, 라벨, 도움말, placeholder, validation rule, 섹션 그룹

### C. 자동 채움(프리필) tool

- 입력: 원문 텍스트(A에서 추출) + Form Schema(B)
- 출력: 필드별 초깃값 + 값의 근거 위치(페이지/문단) + 신뢰도

### D. 사용자 수정 반영(Patch) tool

- 입력: 사용자가 수정한 필드 값(diff) + 기존 필드 값
- 출력: 업데이트된 Form data + 변경 이력(누가/언제/무엇을)

### E. Canonical 구조화 변환 tool

- 입력: 최신 Form data(= “확정된 사실”)
- 출력: 내부 표준 구조(What/Who/How, 기능 분해, 데이터/주체 흐름 등)
- 포인트: 이후 2~6번 에이전트는 **이 표준 구조만** 보게 함

### F. 불확실/충돌 탐지 & 추가 질문 생성 tool

- 입력: Form data + Canonical 구조 + (빈칸/모순/낮은 신뢰도 필드)
- 출력: 추가 확인 질문, “여기 안 채우면 판단 품질 떨어짐” 경고, 우선순위

## 2) 샌드박스 대상성 판단 에이전트

핵심은 “Rule + RAG + 설명 생성”을 **분리**하는 거야.

### A. 대상성 Rule 스크리너 tool

- 입력: (1) 구조화 결과
- 출력: 규제/인허가/실증 필요 가능성 시그널, 키워드/조건 매칭 결과, 1차 라벨(높음/중간/낮음)

### B. 유사 승인 사례 검색(RAG retrieval) tool

- 입력: (1) 구조화 결과 + A의 핵심 시그널
- 출력: 상위 N개 유사 사례(요약, 메타데이터, 유사 이유), “유사 사례 존재 여부” 점수

### C. 근거 정리 & 판정 통합 tool (Decision composer)

- 입력: A 결과 + B 결과
- 출력: 최종 라벨(필요성 높음/병행/낮음), 근거 bullet, “확신/리스크” 표시
- 포인트: **결정 로직을 여기로 고정**(나중에 튜닝 쉬움)

### D. (옵션) 반례/주의사항 생성 tool

- 입력: C 결과
- 출력: “왜 오판할 수 있는지”, 추가 확인 질문, 다음 액션(검토 체크포인트)
- 목적: 컨설턴트 신뢰도 상승(“조심해야 할 것”을 같이 줌)

## 3) 샌드박스 트랙 추천 에이전트

트랙 추천은 “제도 정의 기반 설명”이 핵심이라 **근거 문서 중심**으로 쪼개는 게 좋아.

### A. 트랙 적합도 스코어링 tool

- 입력: (1) 구조화 결과 + (2) 대상성 판단 요약(있으면)
- 출력: 신속확인/실증특례/임시허가 각각 점수 + 점수 근거(조건 충족 여부)

### B. 트랙 정의/요건 RAG tool

- 입력: 후보 트랙 1~2개, 구조화 결과의 핵심 포인트
- 출력: 해당 트랙의 정의/요건 관련 근거 스니펫(출처 포함), 적용 논리 연결점

### C. 추천 사유 생성 tool (Explainable recommender)

- 입력: A 점수 + B 근거
- 출력: 추천 트랙 1순위(+차선), “왜 이 트랙인지” 5~7줄 요약, “고객사 설명용” 문장 템플릿

### D. (옵션) 트랙별 준비 항목 도출 tool

- 입력: 추천 트랙
- 출력: 고객사에게 요청할 추가자료/준비 체크(실증 계획, 안전/책임, 데이터 처리 등)

## 4) 신청서 초안 생성 에이전트

여긴 “글쓰기 에이전트”라기보다 **템플릿 조립 엔진**에 가까워.

### A. 양식 선택/버전 매칭 tool

- 입력: 부처/트랙/프로그램(예: ICT융합/스마트시티/모빌리티) + 신청 타입
- 출력: 사용할 템플릿 ID, 섹션 목록, 필수 필드 스키마

### B. 섹션별 컨텐츠 매핑 tool

- 입력: (1) 구조화 결과 + (2)(3) 판단/추천 결과 + 템플릿 스키마
- 출력: 섹션별 “채워넣을 데이터” JSON (문단 초안 전 단계)
- 포인트: **여기서 출처 메타데이터를 같이 달기**(나중에 6번/7번의 기반)

### C. 섹션 문장 생성 tool

- 입력: B의 섹션별 데이터
- 출력: 섹션별 초안 문단(톤/형식 맞춤), 표/리스트 포함 가능

### D. 일관성/중복/형식 검수 tool

- 입력: 전체 초안
- 출력: 용어 통일, 중복 제거, 누락 섹션 경고, 문장 품질 개선 포인트

### E. 문서 렌더링 tool (DOCX/PDF)

- 입력: 최종 초안 + 템플릿 스타일(회사 로고/서식 등)
- 출력: DOCX/PDF 파일
- 포인트: 이 tool은 LLM이 아니라 **문서 생성 모듈**로 두는 게 안정적

## 5) 전략 추천 에이전트 (고도화)

이건 “유사 사례 → 승인 포인트 패턴 → 이번 건 적용” 파이프라인이 핵심.

### A. 유사 사례 군집/선정 tool

- 입력: (1) 구조화 결과 + (3) 트랙 후보
- 출력: 유사 사례 Top N + “왜 유사한지”(비교 축)

### B. 승인 포인트 패턴 추출 tool

- 입력: 유사 사례 N개(원문 요약/핵심 문장)
- 출력: 반복되는 승인 포인트(실증 범위, 안전/책임, 소비자 고지 등) + 빈도/중요도

### C. 이번 건 적용 전략 생성 tool

- 입력: (1) 구조화 결과 + B의 패턴
- 출력: 이번 건에 맞는 “강조해야 할 포인트”, “피해야 할 서술”, “구체 문장 가이드”

### D. 근거 스니펫 & 인용 후보 tool

- 입력: C 결과 + 사례 원문
- 출력: 참고할 표현(짧은 문장 단위), 출처/사례 ID, 적용 위치 추천
- 포인트: 컨설턴트가 “복붙 가능한” 형태로 주면 체감 가치 큼

### E. (옵션) 유사 사례 없음 대응 tool

- 입력: 검색 결과 없음 신호
- 출력: “근접 분야 사례”로 대체 탐색 전략 + 일반 승인 프레임(보수적 가이드)

## 6) 체크리스트 & 리스크 알림 에이전트 (고도화)

이건 “심사관 관점의 QA”라서, **체크 항목 생성 / 검출 / 개선안**을 분리하는 게 좋아.

### A. 기준 체크리스트 생성 tool

- 입력: 트랙/부처/양식 + (1) 구조화 결과
- 출력: 필수 항목 체크리스트(항목ID, 기준, 합격 조건, 증빙 예시)

### B. 초안 대비 누락/약점 탐지 tool

- 입력: 신청서 초안 + A 체크리스트
- 출력: 누락 항목, 약한 항목(근거 부족/모호/과장), 위험도(상/중/하) + 근거 위치

### C. 리스크 시나리오 생성 tool

- 입력: B 결과
- 출력: 예상 반려/보완 요청 포인트(질문 형태), “왜 위험한지” 설명

### D. 개선 문장/대체 표현 생성 tool

- 입력: (초안의 문제 문장 + B/C 결과) + (5) 전략 추천 결과(있으면)
- 출력: 수정 제안 문장(전/후), 추가로 넣을 근거 항목, 표현 톤 가이드

### E. 최종 검수 리포트 생성 tool

- 입력: A~D 결과
- 출력: 컨설턴트 제출용 리포트(요약, 우선순위, 바로 고칠 것 TOP5)
