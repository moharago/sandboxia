# SandboxIA 중간 보고 발표 자료

## 팀: 4th Law SandboxIA (AI Camp 4기)

---

## 1. Why: 왜 이 서비스를 써야 하는가?

### 1-1. 타겟

**규제샌드박스 신청 컨설턴트**

- 사업자가 규제샌드박스 3가지 트랙(신속확인 / 임시허가 / 실증특례)에 신청하기 **전**, 신청서 작성을 대행하거나 자문하는 전문가
- 대상 사용자: 공공기관 컨설턴트(ICT 융합 지원기관)와 민간 행정사

---

### 1-2. 배경: 신청 과정에서의 컨설턴트

#### 제도의 의의와 목적

- **규제 샌드박스**는 신산업 제품이나 서비스에 대해 일정 조건 하에 **기존 규제를 면제·유예하는 제도**
- 미래 기술을 활용하고자 하는 비즈니스의 실증과 법제화를 가능하게 하며, 국민 일상의 변화와 혁신 기술을 통한 국가 경제 성장을 견인하는 핵심 장치
- 기업은 **3가지 트랙**(신속확인 / 임시허가 / 실증특례) 중 자사 상황에 맞는 경로를 선택해 신청

#### 현황 및 정부의 개선 의지

- 현재, 정부는 규제 샌드박스의 실효성을 높이기 위해 규제 절차 간소화와 불필요 규제 타파에 집중
- 특히 **2026년**부터는 **산업융합촉진법 개정안**으로 창업·R&D·공공조달 등 중소기업 지원 제도가 대폭 손질되어 행정 부담이 완화되며, 이에 따라 **기업들의 규제샌드박스 신청 수요는 더욱 급증**할 것으로 전망

---

### 1-3. 문제 정의

#### 1. 신청서 작성의 복잡성

각 트랙별로 요구하는 신청서 양식과 필수 기재 항목이 다르고, 특히:

- **신속확인**: 규제 법적 논점 + 소관부처 특정 필요
- **임시허가**: 규제 법령 체계 정리 + 면제/완화 범위 구체화 + 안전성·이용자보호방안
- **실증특례**: 위 내용 + 세부 실증계획(기간, 장소, 규모, 방법 등)

→ 컨설턴트는 **의뢰 기업의 기술·서비스를 정확히 이해**한 뒤, **관련 규제 법령을 찾고**, **승인 가능성 높은 논리 구조로 신청서를 구성**해야 함

#### 2. 유사 승인 사례 조사의 비효율

- 과거 승인 사례(PDF)에서 의뢰 사업과 유사한 케이스를 **수작업으로 검색·분석**
- 어떤 법령이 쟁점이었고, 어떻게 논리를 전개했는지 일일이 파악해야 함
- **정보가 파편화**되어 있어 누락·오류 발생 가능

#### 3. 수요 급증 vs. 리소스 한계

- 2026년 산업융합촉진법 개정으로 규제샌드박스 활용 수요 증가 예상
- 컨설턴트 1인당 처리해야 할 케이스는 늘어나지만, 조사·작성에 드는 시간은 그대로
- **퀄리티 유지와 속도 양립이 어려움**

## 2. What: 무엇으로 구성되고, 핵심 가치는 무엇인가?

### 2-1. 서비스 정의

> 규제샌드박스 3개 트랙(신속확인·임시허가·실증특례) 신청서 작성을 지원하는 컨설턴트 전용 워크플레이스

ICT 규제 샌드박스 컨설팅 업무 지원을 위한 AI 에이전트 기반 서비스 고도화가 목표.

본 서비스는 **PDF로 공개된 과거 승인 사례를 구조화된 데이터로 전환**하여, 의뢰 기업과 유사한 사례를 즉시 도출하고, 해당 사례의 **쟁점 법령, 소관부처, 승인 논리**를 참조하여 컨설턴트가 신청서를 빠르고 정확하게 작성하도록 지원한다.

### 2-2. 핵심 가치 (Core Value)

| 가치 | 설명 |
|------|------|
| **유사 사례 즉시 매칭** | 의뢰 기업의 기술·서비스 키워드 입력 시, 과거 승인 사례 DB에서 유사 케이스를 자동 추출 |
| **법령·논점 자동 연결** | 해당 사례에서 쟁점이 된 규제 법령, 소관부처, 논리 구조를 한눈에 제공 |
| **트랙별 신청서 초안 생성** | 신속확인 / 임시허가 / 실증특례 양식에 맞춰 신청서 초안을 자동 구성 |
| **근거 기반 작성** | 유사 승인 사례를 인용·참조하여 설득력 있는 신청서 논리 전개 지원 |

### 2-3. 기대 효과 및 비전

본 솔루션은 컨설턴트가 **단순 반복적인 사례 조사와 양식 작성에서 벗어나**, 의뢰 기업에 최적화된 **전략적 자문에 집중**할 수 있는 환경을 제공한다.

이는 **신청서 작성의 시간 효율성과 정확도를 동시에 개선**하며, 결과적으로 혁신 기업들이 규제 샌드박스라는 기회를 온전히 누려 대한민국 경제 성장에 기여하도록 뒷받침할 것이다.

### 2-4. 목표 (KPI)

- 1건 Case 처리 초기 분석 시간 **50% 이상 단축**
- 초안 생성 기능 사용률 **70% 이상**
- 반복 사용 컨설턴트 비율(재방문) **40% 이상**

## 3. How: 어떻게 사용하는가? (사용자 시나리오)

### 유저 플로우 요약

> 로그인 → Case 생성 → 고객사 서비스 설명 정리 → 샌드박스 대상성 판단 → 트랙 추천 → 신청서 초안 생성 → Case 관리

---

### Step 0. 서비스 진입

**컨설턴트 상태:**
- 고객사 미팅 직후
- 신청서, 미팅 노트, 서비스 설명이 비정형 상태
- 샌드박스 대상이 되는지부터 정리해야 하는 상황

**UI/UX:**
- 기존 Case 대시보드 (고객사명 / 상태 / 마지막 수정일)
- 새 Case 생성

→ Step 1로 이동

---

### Step 1. Case 생성 & 기본 정보 입력

**컨설턴트 입력:**
- Case 이름 (예: A사 헬스케어 AI 서비스)
- 고객사명
- 산업군 (선택)
- 컨설턴트 메모 (선택)

**의미:**
- 이후 모든 분석 결과는 이 Case에 귀속
- 히스토리 관리의 최소 단위

→ Step 2로 이동

---

### Step 2. 서비스 설명 입력 및 구조화

**컨설턴트 입력:**
- 신청서 업로드 (HWP)
- 미팅 노트, 메모는 그대로 붙여넣기

**AI 처리:**
- 신청서, 비정형 텍스트 → 규제 판단 가능한 구조로 변환

**AI 출력:**
- 서비스 요약
- 핵심 기능 분해 (기능 단위) — 규제 판단 관점의 설명(기술 요소, 서비스 제공 방식, 주체)

**UI/UX:**
- AI 출력된 각 항목을 컨설턴트가 수정 가능

→ Step 3으로 이동

---

### Step 3. 샌드박스 대상성 1차 판단

**AI 입력:** Step 1, 2에서 구조화된 고객사 서비스 정보

**AI 출력:**
- 샌드박스 검토 필요성 레벨
  - ✔️ 필요성 높음
  - ⚠️ 기존 제도 병행 검토
  - ❌ 대상 가능성 낮음
- 판단 근거 요약 (규제 공백, 불명확성, 신기술 여부 등)

**UI/UX:**
- "법적 결론 아님" 명확히 표시 (참고용임을 명시)
- 내부 검토용 코멘트 추가 기능

→ ✔️/⚠️인 경우 → 트랙 추천으로 자연 이동
→ ❌인 경우 → Case 상태를 "보류"로 저장 가능

---

### Step 4. 샌드박스 트랙 추천

**AI 입력:** 대상성 판단 결과 + 서비스 기능 구조

**AI 출력:**
- 추천 샌드박스 유형 (1~2개) — 예: 실증특례 / 임시허가
- 추천 사유 — 서비스 특성과 트랙 요건의 매칭 설명
- 부서 기준

**UI/UX:**
- 트랙 선택 확정
- 내부 전략 메모 추가
- 유사 승인 사례와 전략 요약 참조 패널

→ Step 5로 이동

---

### Step 5. 신청서 초안 자동 생성

**AI 입력:** 구조화된 서비스 설명 + 선택된 트랙 + 해당 부서 신청서 템플릿

**AI 출력:**
- 샌드박스 신청서 초안 (신청서 템플릿에 따름)
  - 서비스 개요
  - 기술·서비스 설명
  - 실증 필요성
  - 기대 효과 등

**UI/UX:**
- 신청서 초안 수정 기능
- "AI 작성 / 컨설턴트 검수" 모드 구분
- 수정 이력 저장
- PDF / DOCX 다운로드

→ 완료 → 대시보드 복귀

## 4. 주요 기능 상세

### 기능 1: 서비스 설명 구조화 & 규제 쟁점 정리 (Agent 1 - Service Structurer)

**기능 목적:**
- 컨설턴트가 비정형 입력을 '판단 가능한 구조'로 바꾸는 단계
- 이후 모든 기능(대상성 판단, 트랙 추천, 신청서 생성)의 **기초 데이터**

**입력:**
- 기업의 컨설팅 신청서 (HWP)
- 미팅 노트, 메모 (자유 텍스트)

**출력:**
- **서비스 요약** (LLM 기반) — 무엇을, 누구에게, 어떻게 제공하는지
- **핵심 기능 분해** (기능 단위) — 어떤 행위가 발생하는지(데이터 수집, 자동 판단, 제3자 제공, 물리적 행위 등)
- **잠재 규제 쟁점 후보 리스트** — 사례·법령·규제 데이터(3개 RAG 툴) 기반
  - 유형별 제시 (인허가, 실증 필요성 / 안전, 책임 / 개인정보, 데이터)
  - 구체 법령명 후보 제시 (예: 전자상거래법, 의료기기법, 여신전문금융업법 등)

**데이터 흐름:**
```
[사용자 입력 + HWP 파일]
  → POST /api/v1/agents/structure (FormData)
  → HWP 파싱 (olefile + 자체 패턴 매칭)
  → 문서 병합
  → LLM 구조화 (GPT-4o-mini)
  → CanonicalStructure JSON 생성
  → Supabase 저장 (projects.canonical + projects.application_input)
  → Supabase Storage 파일 업로드
  → 클라이언트 응답
```

**출력 구조 (Canonical Structure):**
```json
{
  "company":    { "company_name": "드론에어", "representative": "...", ... },
  "service":    { "what_action": "드론 배송", "target_users": "개인/기업", ... },
  "technology": { "core_technology": "자율비행 드론", ... },
  "regulatory": { "related_regulations": [...], "regulatory_issues": [...] },
  "metadata":   { "data_completeness": 0.85, "confidence_scores": {...} }
}
```

> **설계 핵심:** `application_input`(원본 보존) + `canonical`(AI 분석용 통일 구조) + `application_draft`(편집용) — 3종 분리 구조. 입력 품질 편차 대응을 위해 구조화 실패 시 추가 정보 요청 프롬프트 발동.

### 기능 2: 샌드박스 대상성 판단 보조 (Agent 2 - Eligibility Evaluator)

**기능 목적:**
- 기업의 서비스가 샌드박스를 검토할 가치가 있는지 판단 제시
- 정확한 판단보다 **컨설턴트가 검토할 수 있는 근거 생성**이 목표

**판단 로직 (Rule + RAG + LLM):**
- **Rule 기반:** 키워드, 조건 매칭으로 1차 필터
- **RAG 기반:** 유사 승인 사례 존재 여부 (R1 + R2 + R3)
- **LLM 추론:** 참고용 설명 생성

**6-Node 워크플로우:**
```
screen_node (키워드 스크리닝, 도메인 태깅)
  → search_regulations_node (R1 RAG: 규제 절차)
  → search_cases_node (R2 RAG: 승인 사례)
  → search_laws_node (R3 RAG: 도메인 법령)
  → compose_decision_node (LLM 종합 판단)
  → generate_evidence_node (근거 데이터 포맷팅)
```

**출력:**
- **샌드박스 검토 필요성:** 필요성 높음 / 기존 제도 검토 병행 / 대상 가능성 낮음
- **판단 근거 요약** (evidence_data)
  - `judgment_summary[]` — 법령/사례/규제 기준별 판단 근거
  - `approval_cases[]` — 유사 승인 사례 (트랙, 기업, 유사도, 상세)
  - `regulations[]` — 관련 법령 (카테고리, 제목, 요약, 출처 URL)
- **바로 출시 시 리스크** (`direct_launch_risks`)
- **신뢰도** (`confidence_score`: 0~100)
- **컨설턴트 최종 선택** (`final_eligibility_label`: required / not_required)

> **핵심:** AI는 보조 역할이며 법적 결론이 아님. 최종 판단은 반드시 컨설턴트가 직접 선택.

### 기능 3: 샌드박스 트랙 추천 + 근거 (Agent 3 - Track Recommender)

**기능 목적:**
- 컨설턴트가 고객사에 '왜 이 트랙인지' 설명할 수 있게 지원
- 추천 사유는 **샌드박스 제도 정의 기반**으로 답변

**4-Node 워크플로우:**
```
retrieve_cases_node (R2 RAG: 유사 사례 검색)
  → score_all_tracks_node (LLM: 3개 트랙 점수 산출)
      - 실증특례: 5개 기준 평가
      - 임시허가: 5개 기준 평가
      - 신속확인: 4개 기준 평가
  → retrieve_definitions_node (R1 RAG: 트랙 정의 검색)
  → generate_recommendation_node (LLM: 최종 추천 + 근거 생성)
```

**출력:**
- **추천 트랙:** 신속확인 / 실증특례 / 임시허가 (1~2개)
- **추천 사유 요약**
- **트랙별 비교표** (track_comparison):

| 항목 | 실증특례 | 임시허가 | 신속확인 |
|------|----------|----------|----------|
| 적합도 | 90점 | 65점 | 40점 |
| 순위 | 1위 | 2위 | 3위 |
| 상태 | AI 추천 | 조건부 가능 | 비추천 |
| 사유 | 긍정/부정/중립 | 긍정/부정/중립 | 긍정/부정/중립 |

## 6. 공유 RAG 시스템 상세

3명의 팀원이 각각 1개의 RAG 도구를 담당하여 구축했다. 모든 에이전트가 공용으로 사용하는 핵심 인프라이다.

### 공통 아키텍처

```
데이터 수집 (collect 스크립트)
  → 청크 분할 + 메타데이터 태깅
  → OpenAI text-embedding-3-small 임베딩
  → ChromaDB 컬렉션 저장
  → 에이전트에서 유사도 검색 + 메타데이터 필터링 호출
```

**Vector Store 추상화:** `BaseVectorStore`(ABC) → `ChromaVectorStore`(구현). LRU 캐시로 컬렉션당 단일 인스턴스 유지.

### R1. 규제제도 & 절차 RAG (`regulation_rag.py`)

**담당:** (팀원 A)

**데이터 소스:**
- ICT 규제 샌드박스(과기부) 관련 제도 문서를 **구조화 JSON**으로 정리
- Google Drive 또는 로컬 JSON(`data/r1_data/r1_rag_ict_only.json`)에서 로드

**데이터 구조:**
- 문서별: `title`, `content`, `category`, `track`, `ministry`, `source_url`
- **10개 카테고리**: 제도정의, 트랙정의, 절차, 제출요건, 심사기준, 트랙비교, 부처별차이, 제도변경, 법적근거, 일반
- **4개 트랙**: 공통, 신속확인, 실증특례, 임시허가
- **6개 부처**: 과기부, 산업부, 금융위, 중기부, 국토부, 환경부

**청킹 전략:**
- 문서 항목 단위 (1 item = 1 chunk)
- 포맷: `[{title}]\n\n{content}`

**검색 기능:**
- `search_regulation(query, track, category, ministry, top_k)` — 유사도 + 메타데이터 3중 필터 (track + category + ministry, `$or`/`$and` 조합)
- `get_track_definition(track)` — 트랙 정의 조회
- `get_application_requirements(track)` — 제출요건 조회
- `get_review_criteria(track)` — 심사기준 조회
- `compare_tracks()` — 트랙 비교 정보
- **Relevance Threshold:** 0.32 이하 결과 필터링

**사용 에이전트:** Agent 2 (Eligibility Evaluator), Agent 3 (Track Recommender)

### R2. 승인 사례 RAG (`case_rag.py`)

**담당:** (팀원 B)

**데이터 소스:**
- 규제 샌드박스 포털에서 공개된 **승인/반려 사례 PDF**를 구조화 JSON으로 변환
- Google Drive 또는 로컬 JSON(`data/r2_data/cases_structured.json`)에서 로드

**데이터 구조:**
```json
{
  "case_id": "실증특례_100_에이아이포펫",
  "track": "실증특례",
  "common_info": {
    "service_name": "서비스명",
    "service_description": "서비스 설명",
    "current_regulation": "현행 규제",
    "special_provisions": "특례 내용",
    "conditions": ["조건1", "조건2"],
    "pilot_scope": "실증 범위",
    "expected_effect": "기대 효과"
  },
  "companies": [{ "company_name": "기업명", "designation_number": "지정번호" }]
}
```

**청킹 전략:**
- 사례 단위 (1 case = 1 chunk)
- 서비스명, 기업명, 설명, 특례 내용, 규제, 기대효과, 실증범위 등을 합쳐서 하나의 텍스트로 구성

**검색 기능:**
- `search_case(query, track, top_k, deduplicate)` — 유사도 검색 + 트랙 필터
- `get_similar_cases_for_application(service_description, track, top_k)` — 유사 사례 추출 + 공통 조건 분석 (Counter 기반 상위 10개)
- `get_approval_patterns(domain_keyword, track, top_k)` — 도메인별 승인 패턴 분석
- `get_case_detail(case_id)` — JSON에서 직접 상세 조회
- **중복 제거:** case_id 기반 자동 Deduplication (3배수 fetch 후 정리)
- **인메모리 캐싱:** 첫 접근 시 전체 JSON 로드 후 글로벌 캐시

**사용 에이전트:** Agent 2 (Eligibility Evaluator), Agent 3 (Track Recommender)

### R3. 도메인별 규제·법령 RAG (`domain_law_rag.py`)

**담당:** (팀원 C)

**데이터 소스:**
- **법령정보 공공 API** (law.go.kr)에서 실시간 수집
- 대상 법률 **11개, 6개 도메인**:

| 도메인 | 법령 |
|--------|------|
| 의료/헬스케어 | 의료법 |
| 금융/핀테크 | 전자금융거래법, 신용정보법 |
| 데이터 | 데이터산업진흥법 |
| 개인정보 | 개인정보보호법 |
| 통신/ICT | 전기통신사업법, 정보통신융합활성화법 |
| 규제/제도 | 산업융합촉진법, 금융혁신지원특별법, 규제자유특구법, 행정규제기본법 |

**데이터 수집 프로세스:**
1. 법령정보 API로 법률 검색 (`/DRF/lawSearch.do`)
2. 법률 상세 조회 (`/DRF/lawService.do`)
3. 조문(Article) → 항(Paragraph) → 호(Subparagraph) → 목(Item) 계층 파싱

**청킹 전략 (항 단위):**
```
법령
  └─ 조 (Article, 예: 제34조)
      └─ 항 (Paragraph, 예: ①, ②) ← [CHUNK 단위]
          └─ 호 (Subparagraph, 예: 1., 2.)
              └─ 목 (Item, 예: 가., 나.)
```
- 항이 있으면 → **항 단위 청킹** (법령명 + 조 헤더 + 항 내용 + 하위 호/목 포함)
- 항이 없으면 → **조 단위 청킹**
- 원숫자(①②③...⑳), 아라비아숫자(1, 2, 12) 등 다양한 항 번호 포맷 파싱 지원

**메타데이터:**
- `law_name`, `article_no`, `article_title`, `paragraph_no`, `chunk_type`(paragraph/article), `domain`, `domain_label`, `ministry`, `enforcement_date`, `citation`

**검색 기능:**
- `search_domain_law(query, domain, top_k)` — 유사도 검색 + 도메인 필터
- 도메인 정규화: 한글 키워드(의료, 금융, 핀테크, 데이터 등) → 영문 코드 자동 매핑

**사용 에이전트:** Agent 2 (Eligibility Evaluator)

```
                      │ canonical + eligibility 전달
                      ▼
Step 3: 트랙 추천
┌─────────────────────────────────────────────────────┐
│ INPUT:  projects.canonical                           │
│ PROCESS: 사례검색(R2) → 3개트랙 LLM 스코어링 →      │

```

**핵심 데이터 구조 3종:**
1. `application_input` — 사용자 원본 (양식 구조 그대로 보존, 수정 안 함)
2. `canonical` — AI 분석용 통일 구조 (에이전트 간 공유 입력)

## 11. Q&A 예상 질문

**Q1. 왜 LangGraph를 선택했나요?**
A: 6개의 전문 에이전트가 각각 다른 도구(RAG, LLM, 파서)를 사용하며 순차적/병렬적으로 협업해야 하는 구조라서, Supervisor 패턴을 자연스럽게 구현할 수 있는 LangGraph를 선택했습니다. 각 에이전트를 StateGraph로 정의하고, 노드 간 상태 전달(TypedDict)이 명확합니다.

**Q2. RAG 데이터는 어떻게 수집했나요?**
A: R1은 규제제도 문서를 구조화 JSON으로 정리, R2는 샌드박스 포털의 승인 사례 PDF를 파싱·구조화, R3는 법령정보 공공 API에서 11개 법률을 실시간 수집했습니다. 3명이 각각 1개씩 담당하여 구축했고, 공통으로 ChromaDB + OpenAI text-embedding-3-small을 사용합니다.

**Q3. Canonical Structure란 무엇인가요?**
A: 다양한 형태의 입력(HWP 신청서, 메모, 텍스트)을 에이전트들이 공통으로 사용할 수 있는 통일된 JSON 구조로 변환한 것입니다. company, service, technology, regulatory 4개 영역으로 나뉘며, Agent 2~6이 이 구조를 입력으로 사용합니다.

**Q4. AI 결과의 신뢰성은 어떻게 보장하나요?**
A: 3가지 장치가 있습니다. (1) confidence_score(0~100)로 AI 자체 신뢰도를 표시, (2) evidence_data로 판단 근거(법령/사례/규제)를 명시적으로 제공, (3) 최종 판단은 반드시 컨설턴트가 직접 선택(final_eligibility_label)합니다. AI는 "보조" 역할이며 법적 결론이 아님을 UI에서 명확히 합니다.

**Q5. 현재까지 가장 어려웠던 기술적 도전은?**
A: (1) HWP 파일 파싱 — 비표준 포맷이라 olefile 기반 자체 파서 구현 필요, (2) Agent 간 데이터 일관성 유지 — Canonical Structure 설계에 여러 차례 반복이 있었음, (3) RAG 검색 품질 — 도메인 특화 데이터에서 의미 있는 유사도 검색을 위한 쿼리 최적화, (4) 법령 API 파싱 — 조/항/호/목 계층 구조와 다양한 항 번호 포맷(①②, 1., 1항 등) 처리.
