## 공용 RAG tool (전 에이전트 공통 사용 권장)

### R1. 규제제도 & 절차 RAG tool

- 데이터: 제도 정의(신속확인/실증특례/임시허가), 절차, 제출 요건, 심사 포인트, 부처별 차이
- 주 사용처: 2(대상성 근거 보강), 3(트랙 추천 근거), 4(신청서 섹션 작성 가이드), 6(체크리스트 기준)

### R2. 승인 사례 RAG tool

- 데이터: 승인/반려 사례, 요약, 조건/부대조건, 실증 범위, 안전·책임, 소비자 고지, 적용 특례
- 주 사용처: 2(유사 사례 존재), 5(전략 패턴 추출), 6(리스크/보완 포인트), (보조로 4 초안 문장 근거)

### R3. 도메인별 규제·법령·인허가 RAG tool

- 데이터: 분야별 주요 법령/인허가 체계(의료/금융/모빌리티 등), 규제 쟁점 유형별 레퍼런스
- 주 사용처: 1(쟁점/법령 후보), 2(대상성 근거), 6(리스크 탐지), (보조로 4 문장 정확도)

> 공용으로 두되, 내부적으로는 하나의 Retrieval API(tool) 아래에서 corpus=制度/사례/도메인법령처럼 라우팅해도 되고, 위처럼 3개 tool로 분리해도 돼. (운영/튜닝 편의는 3개 분리가 보통 좋음)

## 공용 Utility (Tool이 아닌 함수/클래스)

> C0, C1, C2는 LLM이 호출하는 Tool이 아니라, 코드에서 직접 호출하는 유틸리티입니다.

### C0. Evidence/인용 관리 (클래스)

- 모든 RAG 결과에 대해 출처(문서ID/페이지/단락)와 스니펫을 표준 포맷으로 저장/반환
- 주 사용처: 1~6 전부 (특히 4/6에서 강력)
- **타입**: `EvidenceStore` 클래스

### C1. Canonical 구조 (데이터 모델 + 서비스)

- HWP 상담신청서 → 표준 JSON 변환 (pyhwpx, 타이틀 기반 파싱)
- 주 사용처: API 업로드 시 서비스에서 생성, 2~6는 State 입력으로 사용
- **타입**: `CanonicalStructure` Pydantic 모델 + `hwp_parser` 서비스

### C2. Patch/Merge (함수)

- 사용자 수정 diff를 반영하고 변경 이력 남김
- 주 사용처: 1(구조화), 4(초안 버전), 6(수정안 반영)
- **타입**: `apply_patch()`, `merge_patches()` 함수

## 에이전트별 전용 tool 정리

## 1) 기업 서비스 구조화 에이전트 (전용)

- A. 신청서 템플릿/구조 파서
- B. UI Form Schema 생성
- C. 자동 채움(프리필)
- D. 사용자 수정 반영(Patch) _(공용 Patch를 호출해도 됨)_
- F. 불확실/충돌 탐지 & 추가 질문 생성
- G. 규제 쟁점 후보 도출 _(여기서 R3를 많이 씀)_

**공용 RAG 사용:** R3(도메인 법령), (필요 시) R1(제도 정의)

---

## 2) 샌드박스 대상성 판단 에이전트 (전용)

- A. Rule 스크리너(키워드/조건/신호)
- C. 판정 통합(Decision composer)
- D. 반례/주의사항 생성(옵션)

**공용 RAG 사용:** R2(유사 승인사례), R1(제도/절차 근거), R3(법령/인허가 근거)

---

## 3) 샌드박스 트랙 추천 에이전트 (전용)

- A. 트랙 적합도 스코어링
- C. 설명 생성(고객사 설명용 템플릿)
- D. 트랙별 준비 항목 도출(옵션)

**공용 RAG 사용:** R1(트랙 정의/요건), (보조로) R2(사례 기반 설득), R3(도메인 제약)

---

## 4) 신청서 초안 생성 에이전트 (전용)

- A. 양식 선택/버전 매칭
- B. 섹션별 컨텐츠 매핑(“채워넣을 데이터 JSON”)
- C. 섹션 문장 생성
- D. 일관성/중복/형식 검수
- E. DOCX/PDF 렌더링

**공용 RAG 사용:** R1(섹션 요구/작성 가이드), (상황 따라) R2/R3(근거 문장/리스크 표현)

---

## 5) 전략 추천 에이전트 (전용, 고도화)

- B. 승인 포인트 패턴 추출
- C. 이번 건 적용 전략 생성
- D. 인용 후보/표현 추천 생성
- E. 유사 사례 없음 대응

**공용 RAG 사용:** R2가 핵심, (보조로) R1/R3

---

## 6) 체크리스트 & 리스크 알림 에이전트 (전용, 고도화)

- A. 기준 체크리스트 생성
- B. 누락/약점 탐지
- C. 리스크 시나리오 생성
- D. 개선 문장/대체 표현 생성
- E. 최종 검수 리포트 생성

**공용 RAG 사용:** R1(요건/절차 기준), R3(법령 리스크), R2(반려/보완 패턴)
