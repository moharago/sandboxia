# R3 RAG 성능 평가 실험 설계

## 1. 청킹 전략

### 평가 변수

- **청킹 단위:** 법령을 어떤 구조 단위(조 / 항 / 호 등)로 나눠서 벡터화할지 결정하는 기준
- **멀티단위 (멀티 그레뉼러리티):** 같은 법령을 서로 다른 크기의 단위(예: 항+호, 조+항+호)로 동시에 인덱싱하는 전략
- **prefix:** chunk 본문 앞에 법령명·조문번호 같은 구조 정보를 텍스트로 붙여 임베딩에 포함시키는 것
- **Top-k:** 검색 시 유사도가 높은 chunk를 몇 개까지 가져올지 정하는 값
- **hybrid:** 구조 단위로 자르되, 너무 길거나 짧으면 토큰 기준으로 추가 분할/병합하는 보정 방식
- **청킹 크기(min/max_tokens):** 하나의 chunk가 가질 최소·최대 토큰 길이를 제한하는 설정
- **overlap:** 인접 chunk 사이에 일부 문장을 겹치게 포함시켜 문맥 단절을 줄이는 방식

#### 주요 비교 분석

**1. 항 단위 vs 조 단위 (C0 vs C4)**

| 지표      | C0 (항 단위) | C4 (조 단위) | 차이           |
| --------- | ------------ | ------------ | -------------- |
| MH-Recall | 0.7000       | **0.8333**   | **+13.33%p** ↑ |
| Recall    | 0.6500       | **0.8000**   | **+15.00%p** ↑ |
| MRR       | 0.6361       | **0.6650**   | +2.89%p ↑      |
| 청크 수   | 3,184        | 1,082        | -66% (효율 ↑)  |

> **결론**: 조 단위 청킹이 항 단위보다 **MH-Recall +13%p, Recall +15%p** 우수. 같은 조의 항들이 경쟁하지 않아 누락 감소.

**2. prefix+hybrid 효과 (C0 vs C1 / C4 vs C5)**

| 비교          | 기본 (C0/C4) | +prefix+hybrid (C1/C5) | MH-Recall 차이 |
| ------------- | ------------ | ---------------------- | -------------- |
| 항 단위       | C0: 0.7000   | C1: 0.7667             | **+6.67%p** ↑  |
| 조 단위       | C4: 0.8333   | C5: 0.8333             | 0%p (동등)     |
| MRR (항 단위) | 0.6361       | 0.6444                 | +0.83%p ↑      |

> **결론**: 항 단위에서는 prefix+hybrid가 효과적(+6.67%p). 조 단위에서는 이미 충분한 문맥 포함으로 추가 효과 없음.

**3. 멀티 그래뉼러리티 효과 (C1 vs C2 / C5 vs C6)**

| 비교    | 단일 그래뉼러리티 | 멀티 그래뉼러리티 | MH-Recall 차이 | 청크 수 증가 |
| ------- | ----------------- | ----------------- | -------------- | ------------ |
| 항 단위 | C1: 0.7667        | C2: 0.7667        | 0%p (동등)     | +84%         |
| 조 단위 | C5: 0.8333        | C6: 0.7667        | **-6.67%p** ↓  | +259%        |

> **결론**: 멀티 그래뉼러리티는 성능 향상 없이 청크 수만 증가. 조 단위에서는 오히려 **성능 하락**.

**4. Top-k 확장 효과 (C2 vs C3 / C6 vs C7)**

| 비교    | K=5        | K=8        | MH-Recall 차이 |
| ------- | ---------- | ---------- | -------------- |
| 항 단위 | C2: 0.7667 | C3: 0.7667 | 0%p (동등)     |
| 조 단위 | C6: 0.7667 | C7: 0.7667 | 0%p (동등)     |

> **결론**: K를 5→8로 늘려도 MH-Recall 변화 없음. 멀티 그래뉼러리티 환경에서 Top-k 증가 효과 미미.

**5. 청크 크기 효과 (C5 vs C8)**

| 지표      | C5 (200/1200) | C8 (100/800) | 차이           |
| --------- | ------------- | ------------ | -------------- |
| MH-Recall | 0.8333        | **0.8667**   | **+3.34%p** ↑  |
| Recall    | 0.8000        | **0.8333**   | **+3.33%p** ↑  |
| MRR       | **0.6411**    | 0.6189       | -2.22%p ↓      |
| 청크 수   | 1,123         | 1,188        | +5.8%          |

> **결론**: 작은 청크(max=800)가 긴 조문 분할로 **검색 커버리지 향상**. 단, 첫 번째 결과 품질(MRR)은 소폭 하락.

#### 최종 권장

| 순위 | 프리셋 | 추천 이유                                                            |
| ---- | ------ | -------------------------------------------------------------------- |
| 1    | **C8** | **최고 성능** (MH-Recall 0.87, Recall 0.83) - 작은 청크로 커버리지 ↑ |
| 2    | C4     | MRR 최고 (0.67), 최소 청크 수 (1,082) - 첫 번째 결과 품질 중요 시    |
| 3    | C5     | C4와 동등 Recall, prefix+hybrid로 청크 정규화 (운영 안정성)          |

#### 목표 대비 Gap (실무 합격선 기준)

| 지표        | 현재 최고 (C8) | 목표    | Gap        |
| ----------- | -------------- | ------- | ---------- |
| MH-Recall@5 | 0.8667         | ≥ 0.90  | **-3.3%p** |
| Recall@5    | 0.8333         | ≥ 0.85  | **-1.7%p** |
| MRR         | 0.6189         | ≥ 0.70  | -8.1%p     |
| Latency P95 | ~320ms         | ≤ 250ms | ⚠️ 초과    |

> **핵심 발견**: C8(작은 청크)이 **MH-Recall 86.67%**로 최고 성능. 목표 90%까지 약 **3.3%p** 남음.
>
> **다음 단계**: 임베딩 모델 변경(text-embedding-3-large), Reranker 추가로 목표 90% 달성 필요.

#### Latency 분석: 조 단위가 느린 이유

**청크 수 vs Latency 비교:**

| 프리셋   | 청크 수 | 평균 청크 길이 | P95   |
| -------- | ------- | -------------- | ----- |
| C1 (항)  | 3,196   | ~176자         | 188ms |
| C4 (조)  | 1,082   | ~540자         | 278ms |
| C8 (조)  | 1,188   | ~481자         | 319ms |

청크 수는 1/3인데 오히려 Latency가 더 김.

**원인:**

1. **청크 텍스트 I/O 오버헤드**: 조 단위 청크는 평균 480~540자 (항 단위의 2.5~3배). Vector DB에서 결과를 가져올 때 더 큰 텍스트 전송 → I/O 증가
2. **후처리 시간**: 검색된 긴 텍스트를 파싱/처리하는 시간 증가
3. **측정 시점 변동**: OpenAI 임베딩 API 호출 (쿼리당 50~100ms)이 네트워크 상태에 따라 변동. P95는 상위 5% 느린 쿼리 기준이라 outlier 영향 큼

**개선 방법:**

| 방법                              | 예상 효과                | 구현 난이도 |
| --------------------------------- | ------------------------ | ----------- |
| **로컬 임베딩 모델** (BGE-M3)     | -50~100ms (API 호출 제거) | 중          |
| **Vector DB 변경** (Qdrant 등)    | -30~50ms                 | 중          |
| **HNSW 인덱스 튜닝**              | -10~20ms                 | 낮음        |
| **쿼리 결과 캐싱**                | 반복 쿼리 시 -200ms+     | 낮음        |
| **메타데이터/본문 분리 저장**     | I/O 감소                 | 중          |

**권장 우선순위:**

1. **단기**: 쿼리 캐싱 + HNSW 파라미터 튜닝 (ef_search 조정)
2. **중기**: 로컬 임베딩 모델로 전환 (BGE-M3-ko)
3. **장기**: Qdrant/Milvus로 Vector DB 마이그레이션

> C8의 319ms는 **임베딩 API 호출 시간(~100ms)**을 제외하면 실제 검색은 ~220ms 수준. 로컬 임베딩만으로도 250ms 목표 달성 가능.

| --------------------- | ---------------------- | ---------------------------- | -------------------- |

## 2. 임베딩 모델

### 고려 요소

- **Retrieval 성능**: 질의와 가장 관련 있는 법령 조항을 Top-k 안에 정확히 포함시키는 능력.
- **한국어 성능 / 도메인 적합성**: 한국어 법령 특유의 표현·문장 구조·법률 용어를 얼마나 정확히 의미적으로 표현하는지에 대한 적합도.
- **벡터 차원 수 (Dimension)**: 문장의 의미를 얼마나 정밀하게 표현할 수 있는지를 결정하는 임베딩 벡터의 길이.
- **입력 길이 한계 (Max Tokens)**: 하나의 조항 또는 긴 법령 문장을 잘리지 않고 안정적으로 임베딩할 수 있는 최대 입력 길이.
- **Query/Passage 구조 지원 여부**: 질의와 문서를 구분해 학습된 구조를 활용해 검색 정밀도를 높일 수 있는지 여부.
- **Cost & Latency**: 임베딩 생성 시 발생하는 토큰 비용과 응답 속도 등 실서비스 운영 효율성에 영향을 주는 요소.
- **운영 구조 (API vs 로컬)**: 외부 API 호출 방식인지, 내부 서버에서 직접 운영 가능한 모델인지에 따른 보안·확장성·관리 복잡도 차이.
