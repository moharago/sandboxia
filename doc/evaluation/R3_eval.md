# R3 RAG 성능 평가 실험 설계

## 1. 청킹 전략

### 평가 변수

- **청킹 단위:** 법령을 어떤 구조 단위(조 / 항 / 호 등)로 나눠서 벡터화할지 결정하는 기준
- **멀티단위 (멀티 그레뉼러리티):** 같은 법령을 서로 다른 크기의 단위(예: 항+호, 조+항+호)로 동시에 인덱싱하는 전략
- **prefix:** chunk 본문 앞에 법령명·조문번호 같은 구조 정보를 텍스트로 붙여 임베딩에 포함시키는 것
- **Top-k:** 검색 시 유사도가 높은 chunk를 몇 개까지 가져올지 정하는 값
- **hybrid:** 구조 단위로 자르되, 너무 길거나 짧으면 토큰 기준으로 추가 분할/병합하는 보정 방식
- **청킹 크기(min/max_tokens):** 하나의 chunk가 가질 최소·최대 토큰 길이를 제한하는 설정
- **overlap:** 인접 chunk 사이에 일부 문장을 겹치게 포함시켜 문맥 단절을 줄이는 방식

| Recall    | 0.6500       | **0.8000**   | **+15.00%p** ↑ |

#### Latency 분석: 조 단위가 느린 이유

**청크 수 vs Latency 비교:**

| 프리셋  | 청크 수 | 평균 청크 길이 | P95   |
| ------- | ------- | -------------- | ----- |
| C1 (항) | 3,196   | ~176자         | 188ms |
| C4 (조) | 1,082   | ~540자         | 278ms |
| C8 (조) | 1,188   | ~481자         | 319ms |

청크 수는 1/3인데 오히려 Latency가 더 김.

**원인:**

1. **청크 텍스트 I/O 오버헤드**: 조 단위 청크는 평균 480~540자 (항 단위의 2.5~3배). Vector DB에서 결과를 가져올 때 더 큰 텍스트 전송 → I/O 증가
2. **후처리 시간**: 검색된 긴 텍스트를 파싱/처리하는 시간 증가
3. **측정 시점 변동**: OpenAI 임베딩 API 호출 (쿼리당 50~100ms)이 네트워크 상태에 따라 변동. P95는 상위 5% 느린 쿼리 기준이라 outlier 영향 큼

**개선 방법:**

| 방법                           | 예상 효과                 | 구현 난이도 |
| ------------------------------ | ------------------------- | ----------- |
| **로컬 임베딩 모델** (BGE-M3)  | -50~100ms (API 호출 제거) | 중          |
| **Vector DB 변경** (Qdrant 등) | -30~50ms                  | 중          |
| **HNSW 인덱스 튜닝**           | -10~20ms                  | 낮음        |
| **쿼리 결과 캐싱**             | 반복 쿼리 시 -200ms+      | 낮음        |
| **메타데이터/본문 분리 저장**  | I/O 감소                  | 중          |

**권장 우선순위:**

1. **단기**: 쿼리 캐싱 + HNSW 파라미터 튜닝 (ef_search 조정)
2. **중기**: 로컬 임베딩 모델로 전환 (BGE-M3-ko)
3. **장기**: Qdrant/Milvus로 Vector DB 마이그레이션

> C8의 319ms는 **임베딩 API 호출 시간(~100ms)**을 제외하면 실제 검색은 ~220ms 수준. 로컬 임베딩만으로도 250ms 목표 달성 가능.

## 2. 임베딩 모델

### 고려 요소

- **Retrieval 성능**: 질의와 가장 관련 있는 법령 조항을 Top-k 안에 정확히 포함시키는 능력.
- **한국어 성능 / 도메인 적합성**: 한국어 법령 특유의 표현·문장 구조·법률 용어를 얼마나 정확히 의미적으로 표현하는지에 대한 적합도.
- **벡터 차원 수 (Dimension)**: 문장의 의미를 얼마나 정밀하게 표현할 수 있는지를 결정하는 임베딩 벡터의 길이.
- **입력 길이 한계 (Max Tokens)**: 하나의 조항 또는 긴 법령 문장을 잘리지 않고 안정적으로 임베딩할 수 있는 최대 입력 길이.
- **Query/Passage 구조 지원 여부**: 질의와 문서를 구분해 학습된 구조를 활용해 검색 정밀도를 높일 수 있는지 여부.
- **Cost & Latency**: 임베딩 생성 시 발생하는 토큰 비용과 응답 속도 등 실서비스 운영 효율성에 영향을 주는 요소.
- **운영 구조 (API vs 로컬)**: 외부 API 호출 방식인지, 내부 서버에서 직접 운영 가능한 모델인지에 따른 보안·확장성·관리 복잡도 차이.

## 3. 벡터 DB

### 고려 요소

- **ANN 알고리즘 및 튜닝 가능 여부**: 근사 최근접 탐색 방식(HNSW, IVF 등)을 사용하며 Recall–Latency 균형을 세밀하게 조정할 수 있는 유연성.
- **Hybrid Search 지원 여부**: Dense 벡터 검색과 BM25 같은 키워드 기반 검색을 결합해 검색 정확도를 보완할 수 있는 기능.
- **Metadata Filter 성능**: 벡터 유사도 검색과 메타데이터 조건 필터링을 동시에 적용해도 성능 저하 없이 정확하게 결과를 반환하는 능력.
- **P95 Latency**: 전체 요청 중 95%가 이 시간 이내에 응답되는 지연 시간으로, 실서비스 체감 속도를 판단하는 핵심 지표.
- **대용량 확장성**: 수백만~수천만 개 이상의 벡터를 저장·검색할 때도 성능 저하 없이 확장 가능한 구조인지 여부.
- **운영 안정성 (Persistence / 장애 대응)**: 데이터 영속성 보장, 장애 발생 시 복구 기능, 백업 및 스냅샷 지원 등 서비스 연속성을 유지하는 능력.

### Hybrid Search 실험 설계

#### 실험 목적

Dense Search(벡터 유사도)만으로 놓치는 **법률 용어/조문 번호 키워드 매칭**을 Sparse Search로 보완하여 Recall 향상 가능성 검증.

#### 평가 변수

- **Sparse 모델**: SPLADE(학습된 희소 임베딩) vs BM25(전통 키워드 빈도)
- **Alpha (Dense 가중치)**: Dense와 Sparse의 결합 비율 (0.0~1.0)
  - alpha=0.7 → Dense 70% + Sparse 30%
  - alpha=0.5 → 균형 (50:50)
  - alpha=0.3 → 키워드 중심 (Dense 30%)

#### 실험 조합

법령 데이터 특성(정형화된 용어, 조문 번호 참조)을 고려한 테스트 조합:

| 프리셋 | Sparse 모델 | Alpha | 설명                     | 테스트 의도                       |
| ------ | ----------- | ----- | ------------------------ | --------------------------------- |
| **H0** | SPLADE      | 0.7   | SPLADE 기본 (Dense 중심) | 기본 성능 확인                    |
| **H1** | SPLADE      | 0.5   | SPLADE 균형              | Dense/Sparse 균형점 탐색          |
| **H2** | SPLADE      | 0.3   | SPLADE 키워드 중심       | 법률 용어 매칭 극대화             |
| **H3** | BM25        | 0.7   | BM25 기본 (Dense 중심)   | SPLADE vs BM25 비교 기준          |
| **H4** | BM25        | 0.5   | BM25 균형                | 전통 키워드 방식 균형점           |
| **H5** | BM25        | 0.3   | BM25 키워드 중심         | 조문 번호/법령명 정확 매칭        |
| **H6** | SPLADE      | 0.9   | Dense 극대화             | Hybrid 효과 하한선 (거의 Dense만) |
| **H7** | BM25        | 0.9   | Dense 극대화 + BM25      | H6과 Sparse 모델 차이 비교        |

#### 실험 결과
