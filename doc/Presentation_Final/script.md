## Slide 1 — 표지 (35초)

안녕하세요, 1st Law SandboxIA 팀입니다.

저희가 만든 서비스는 **SandboxIA**, AI 멀티 에이전트 기반 규제 샌드박스 컨설팅 지원 시스템입니다.

"규제 샌드박스"란, 새로운 기술로 만든 서비스가 기존 규제에 막혀 출시를 못 할 때, 정부에 "이 규제를 한시적으로 풀어달라"고 신청하는 제도입니다. 예를 들어 원격 진료 앱을 만들었는데 의료법상 대면 진료만 인정한다면, "제한된 범위에서 시험해보겠습니다"라고 신청하는 거죠. SandboxIA는 이 신청 과정을 AI가 도와주는 시스템입니다.

## Slide 3 — 기획 배경 & 목적 (1분 30초)

기획 배경입니다. 해결하고자 한 문제는 크게 세 가지입니다.

**첫째, 신청서 작성이 매우 복잡합니다.** 규제 샌드박스에는 신속확인, 임시허가, 실증특례 세 가지 트랙이 있고, 트랙마다 양식, 필수 항목, 법적 논점이 전혀 다릅니다. 컨설턴트는 기업 기술 이해, 관련 법령 조사, 규제 해소 논리 구성을 동시에 해야 하는 복합적인 작업입니다.

**둘째, 유사 사례 조사가 비효율적입니다.** 과거 승인 사례 PDF를 하나하나 수작업으로 검색해야 하고, 어떤 법령이 쟁점이었는지, 어떤 논리로 심사를 통과했는지를 일일이 파악해야 합니다. 정보가 흩어져 있어 파편화 문제도 있습니다.

**셋째, 수요는 늘어나는데 리소스는 제한적입니다.** 2026년 법 개정으로 샌드박스 수요가 증가하고 있지만, 컨설턴트 리소스는 한정적입니다. 사람을 더 뽑거나 도구로 효율을 높이거나 둘 중 하나인데, 저희는 후자를 선택했습니다.

**타겟 사용자**는 ICT융합 지원기관 컨설턴트와 민간 행정사입니다.

## Slide 4 — 프로젝트 목표 & 핵심 가치 (1분)

정량 목표는 **4단계 AI 파이프라인 완성**, **승인 사례 281건 DB 구축**, **3개 RAG 검색 시스템 구축**입니다.

핵심 기능 세 가지입니다. **하나**, 서비스 구조화 → 대상성 판단 → 트랙 추천 → 신청서 초안의 4단계 AI 파이프라인. 실제 컨설턴트의 업무 흐름을 그대로 AI 파이프라인으로 옮겼습니다. **둘**, 승인 사례·법률·제도 문서를 의미 기반으로 검색하는 3-RAG 시스템. 키워드가 정확히 일치하지 않아도 의미가 비슷하면 찾아줍니다. **셋**, 트랙별 양식에 맞춘 신청서 초안 자동 생성과 DOCX 다운로드.

중요한 설계 원칙 — **AI는 보조 역할이고, 최종 판단은 반드시 컨설턴트가 합니다.**

## Slide 5 — 기술 스택 (40초)

기술 스택입니다.

**프론트엔드**는 Next.js 16, React 19, TailwindCSS 4, TanStack Query 5입니다. 신청서 초안 편집에는 서식이 있는 에디터가 필요해서 Tiptap 3 리치 에디터를 도입했습니다.

**백엔드**는 FastAPI + LangGraph 1.0으로 멀티 에이전트를 구현했고, GPT-4o와 text-embedding-3-large를 3개 RAG 공통으로 사용합니다. 벡터 DB는 Qdrant, RAG 평가에는 RAGAS 프레임워크를 활용했습니다.

**인프라**는 Supabase로 DB와 인증을 통합 관리하고, Docker Compose로 컨테이너화, CI/CD는 GitHub Actions, 코드 리뷰는 CodeRabbit AI가 자동 수행합니다.

## Slide 6 — 컨설팅 워크플로우 × AI 파이프라인 (1분 45초)

핵심 서비스 기능입니다. 실제 컨설턴트의 업무 흐름을 그대로 AI 파이프라인으로 설계했습니다.

**Step 1** 서비스 구조화 — 기업이 보낸 HWP 신청서를 파싱해서 통일된 JSON 구조로 변환합니다. 이 구조가 이후 모든 단계의 입력이 됩니다.

**Step 2** 대상성 판단 — "이 서비스가 샌드박스 신청 자격이 있는가?"를 3개 RAG 병렬 검색 후 LLM이 종합 판정합니다. 컨설턴트가 AI 판정을 Override할 수 있도록 설계했습니다.

**Step 3** 트랙 추천 — 세 가지 트랙 중 가장 적합한 트랙을 체크리스트 스코어링과 유사 사례 매칭으로 추천합니다.

**Step 4** 신청서 초안 — 추천 트랙 양식에 맞춰 자동으로 초안을 생성하고, 에디터에서 편집 후 DOCX 다운로드합니다.

오른쪽 **Agent × RAG 매트릭스**를 보시면, Step 1만 RAG를 사용하지 않고, Step 2~4는 R1(제도/절차), R2(승인사례), R3(법령) 세 가지 RAG를 모두 활용합니다. 이 3개 RAG는 에이전트들이 공유하는 인프라로, 팀원 3명이 각각 하나씩 담당 구축했습니다.

## Slide 7 — Step 1: 서비스 구조화 (1분 15초)

Step 1의 핵심은 비정형 HWP를 **Canonical Structure**(통일 JSON)로 변환하는 것입니다. 이 JSON이 제대로 만들어져야 뒤의 판단, 추천, 초안 생성이 정확해집니다.

2-Node 워크플로우입니다.

**Parse HWP** — HWP는 비공개 바이너리 포맷이라 일반적인 방법으로 텍스트를 꺼낼 수 없습니다. olefile로 OLE 컨테이너를 열고, zlib 압축 해제, UTF-16LE 디코딩 후 정규식으로 회사명, 서비스명, 체크박스 상태 등 50개 이상 필드를 추출합니다.

**Build Structure** — 최대 4장 HWP를 8개 필드 그룹으로 분류하고, first-win 전략으로 병합합니다. 같은 필드가 여러 문서에 나오면 먼저 나온 값을 우선 적용합니다. 그 다음 GPT-4o(temp 0.1)가 Canonical JSON으로 변환합니다.

설계 원칙: **체크박스, 서명, 날짜 같은 사실 데이터는 HWP 파서 값을 LLM 출력보다 항상 우선 적용합니다.** AI가 판단할 영역과 그대로 복사할 영역을 명확히 분리했습니다.

---

## Slide 8 — Step 2: 대상성 판단 (1분 30초)

"이 서비스가 샌드박스 신청 자격이 있는가?"를 판단하는 4-Node 워크플로우입니다. 컨설턴트 입장에서 첫 번째 관문이에요. 자격이 안 되는 걸 준비하면 시간 낭비니까요.

**Screen** — 5개 도메인별 키워드 사전으로 빠르게 규제 영역을 파악합니다. LLM 호출 전에 검색 방향을 잡아주는 역할입니다.

**Search All RAG** — R1, R2, R3를 ThreadPoolExecutor로 **병렬 검색**합니다. 순차적으로 하면 시간이 3배 걸리니까 병렬로 처리한 겁니다.

**Compose Decision** — GPT-4o가 종합 판정합니다. 신뢰도는 LLM이 직접 매기는 게 아니라, 근거 수량과 일관성에 기반한 규칙으로 별도 계산합니다. LLM의 자기 확신은 신뢰하기 어렵기 때문입니다.

**Generate Evidence** — 9개 LLM 호출을 병렬 실행해서 각 RAG 항목에 대한 근거 설명을 생성합니다.

핵심 — AI가 "대상 아님"이라 판정해도 컨설턴트가 Override할 수 있습니다. 도메인 전문가의 판단이 항상 우선입니다.

---

## Slide 9 — Step 3: 트랙 추천 (1분 15초)

세 트랙의 차이를 간단히 말씀드리면, **실증특례**는 "안전성 미검증이라 제한 범위에서 테스트", **임시허가**는 "안전성은 검증됐는데 허가 근거가 없어 임시 허가", **신속확인**은 "규제 적용 여부가 불명확하니 빨리 확인"입니다.

4개 노드로 자동화합니다. R2에서 유사 사례를 검색하고, 트랙별 4~5개 체크리스트로 LLM 스코어링을 합니다. R3으로 차단 규제를 탐지하면 최대 -30점 페널티를 부과합니다. 아무리 체크리스트 점수가 높아도 법적으로 명확히 막혀있으면 해당 트랙 순위를 낮춰야 하니까요. 마지막으로 R1 정의 + R2 사례 + R3 법령 + 스코어를 종합해서 트랙별 추천 사유와 근거를 생성합니다. LLM이 인용한 사례가 실제 RAG 결과와 일치하는지 출처 검증 후처리도 수행합니다.

---

## Slide 10 — Step 4: 신청서 초안 생성 (1분 15초)

추천 트랙 양식에 맞춰 초안을 만듭니다. 트랙별로 양식이 다른데, 신속확인은 2장, 임시허가와 실증특례는 각 4장짜리입니다.

핵심은 **5단계 필드 채우기**입니다. "AI가 모든 칸을 쓴다"가 아니라 "써야 할 칸과 복사할 칸을 명확히 구분한다"는 설계 철학입니다.

회사명, 매출액 같은 사실 데이터는 **그대로 복사**(Pass-through), HWP 원문은 **매핑 테이블로 배치**, 트랙이 다르면 **사유 필드 재생성**, 그래도 비어있는 서술형 필드만 **LLM이 작성**합니다. AI가 숫자를 바꾸거나 이름을 틀리게 쓸 위험을 원천 차단한 겁니다.

원본(application_input), AI 분석(canonical), 편집본(application_draft)을 완전히 분리해서, 컨설턴트가 초안을 수정해도 원본은 항상 보존됩니다. Tiptap 에디터에서 편집하고 DOCX로 다운로드합니다.

---

## Slide 11 — 기능 구현 기술 아키텍처 (1분)

전체 아키텍처입니다. Client(Next.js) ↔ Server(FastAPI + LangGraph) ↔ Data(Supabase + Qdrant) 3계층 구조입니다.

클라이언트와 서버 간 일반 요청은 REST, 분석 진행 상황은 SSE 스트림으로 실시간 전달합니다. SSE를 쓴 이유는, Agent 분석이 최대 2분까지 걸리는데 그동안 "지금 어떤 단계를 처리 중입니다"를 보여줘야 하기 때문입니다.

LLM은 GPT-4o, 임베딩은 text-embedding-3-large를 3개 RAG 공통으로 사용합니다. R3 법령 RAG 평가에서 고차원 모델이 10%p 더 높은 성능을 보였고, 운영 환경에서 임베딩 모델을 RAG별로 분리 관리하기 어려워 성능이 더 좋은 E1으로 통일했습니다.

## Slide 12 — RAG 툴 구성 (1분 15초)

3-RAG 공유 아키텍처입니다. 에이전트 성능을 좌우하는 가장 중요한 컴포넌트입니다.

**R1 규제제도 & 절차** — ICT 규제 샌드박스 제도 문서 기반. 10개 카테고리 × 4개 트랙 × 6개 부처 구조로, track + category 필터로 정밀 검색 가능합니다.

**R2 승인 사례** — 281건, 1건=1문서(청킹 없음). 데이터가 최대 1,211자로 임베딩 토큰 한도의 10%도 안 되어 쪼갤 이유가 없었습니다. 정보 부족 50건을 GPT-4o로 보강(enriched)했습니다.

**R3 도메인별 법령** — 국가법령정보센터 API에서 11개 법률 수집, 조 단위 1,188 청크(평균 480자). 차단 규제 탐지에 핵심입니다.

공통으로 Qdrant 벡터 DB를 사용하고, BaseVectorStore 추상 클래스로 persistent/http/ephemeral 세 모드를 지원합니다. 평가할 때는 ephemeral 모드로 운영 DB를 오염시키지 않습니다.

## Slide 13 — R1 RAG 평가 (1분 30초)

RAG 성능 평가 파트입니다. R1은 147개 청크, 30개 질의-정답 쌍으로 3단계 최적화를 수행했습니다.

**1단계 임베딩 모델** — 3종을 비교했는데, E1 text-embedding-3-large가 MH-Recall 73.3%, MRR 88.3%로 성능과 속도 모두 1위였습니다. E2 Solar는 차원이 4096으로 가장 컸지만 성능은 E1보다 낮고 지연시간은 4배. 무조건 높다고 좋은 게 아니라는 걸 확인했습니다.

**2단계 검색 전략** — 하이브리드 검색은 오히려 MRR이 21%p 떨어졌습니다. R1 데이터가 구조화된 제도 문서라 키워드 매칭이 노이즈를 만든 겁니다. 리랭킹은 성능은 올랐지만 지연 129배로 실시간 부적합. **벡터 검색 단독이 가장 효율적**이었습니다.

**3단계 생성 품질** — Faithfulness 98.67%, 트랙 프리픽스 적용으로 MH-Recall 15%p 향상.

최종 확정: **E1 + Vector Search + gpt-4o-mini**

## Slide 14 — R2 RAG 평가 (1분 45초)

R2는 4단계 최적화를 수행했습니다. "비슷한 서비스가 과거에 승인받은 사례를 찾아주는" RAG이라, 검색 품질이 곧 서비스 품질과 직결됩니다.

**1단계 데이터 전략** — 같은 281건을 structured, enriched, hybrid로 비교했습니다. GPT-4o로 핵심 키워드와 기술 설명을 추출해서 보강한 **enriched가 Recall +7.3%p**로 채택. 원본 중 서비스 설명이 너무 짧거나 모호한 게 있었는데, 핵심 정보를 명시적으로 뽑아주니 임베딩이 더 정확해진 겁니다.

**2단계 임베딩 모델** — 4종 비교했는데 모델 간 차이가 ~2.5%p로 소폭. 1건=1문서라 경량 모델로도 충분했습니다. 평가에서는 E0가 MRR 1위였지만, R1/R3과 **임베딩 모델 통일이 필요해서 E1 채택**. 성능 차이가 미미해 손해 없었습니다.

**3단계 생성 모델** — RAGAS로 4개 모델 비교. gpt-4o가 Faithfulness 0.975, Relevancy 0.718, P50 2.9초로 세 지표 모두 1위.

**4단계 구성 튜닝** — top_k를 3, 5, 7, 10으로 비교. k=7은 MH-Recall은 올랐지만 Answer Relevancy가 8.4%p 떨어졌습니다. 관련 없는 문서가 LLM을 혼란시킨 거죠. **k=5가 검색과 생성 품질의 최적 균형점**. Score threshold는 점수 분포가 너무 좁아 유해하므로 사용 안 함.

최종 확정: **enriched + E1 + gpt-4o + top_k=5**

---

## Slide 15 — R3 RAG 평가 (1분 45초)

R3는 "이 서비스가 어떤 법에 막히는가?"를 찾는 RAG으로, 정확도가 특히 중요합니다. 잘못된 법령을 인용하면 신청서 전체의 신뢰도가 무너지니까요.

**1단계 청킹 전략** — 9개 조합을 실험했습니다. 항(paragraph) 단위로 쪼개면 같은 조의 1항, 2항, 3항이 서로 경쟁하면서 정작 필요한 조문이 밀려나는 현상이 있었습니다. **조 단위로 합치니 이 경쟁이 사라지면서 MH-Recall 13%p 향상**. C8(조 단위 + 법률명 프리픽스) 채택.

**2단계 임베딩 모델** — E1이 E0 대비 **MH-Recall 10%p 높음**. 법률 텍스트는 한 문장에 여러 개념이 압축된 "의미 밀도가 높은" 텍스트여서 고차원 임베딩이 효과적이었습니다.

**3단계 하이브리드 검색** — Qdrant + SPLADE가 BM25 대비 **MRR 8~13%p 높음**. 한국어 BM25는 띄어쓰기 기반이라 "원격의료"를 분해 못 하지만, SPLADE는 WordPiece로 서브워드 분해가 가능합니다. 한국어 RAG에서 하이브리드를 할 때 BM25가 아닌 SPLADE가 훨씬 효과적이라는 교훈을 얻었습니다.

최종 확정: **C8 + E1 + Qdrant SPLADE**, MH-Recall 96.7%

## Slide 16 — Agent 아키텍처 (1분)

LangGraph StateGraph로 구현된 4개 Agent 파이프라인입니다.

Agent 1이 **Canonical Structure**를 만들면, Agent 2/3/4가 동일한 JSON을 입력으로 공유합니다. 에이전트마다 데이터를 다르게 해석하는 불일치를 근본적으로 방지한 설계입니다. 새 Agent를 추가하더라도 이 Canonical Structure만 참조하면 됩니다.

각 Agent는 병렬 처리를 적극 활용합니다. Agent 2는 R1/R2/R3 병렬 검색 + 9개 근거 병렬 생성, Agent 3은 트랙별 스코어링 + R3 페널티 반영, Agent 4는 Pass-through와 LLM 생성을 5단계로 조합합니다.

하단 End-to-End Flow처럼, HWP 업로드 → 구조화 → Canonical Structure → 판정 + 추천 + 초안까지 전체 데이터가 일관되게 흐릅니다.

## Slide 17 — 서비스 완성 (시연) (1분 15초)

실제 서비스 시연입니다.

Google OAuth로 로그인하면 대시보드에서 진행 중인 프로젝트와 Step 1~4 진행 상태를 볼 수 있습니다. 새 프로젝트를 만들고 HWP를 업로드하면 Agent 1이 파싱과 구조화를 수행하는데, SSE로 "HWP 파싱 중...", "구조화 중..." 같은 진행 상태가 실시간 표시됩니다.

구조화 완료 후 Step 2 대상성 판단, Step 3 트랙 추천, Step 4 신청서 초안 편집과 DOCX 다운로드까지 이어집니다.

**AI Agent 4개, RAG 3개, REST + SSE API, 대시보드 + Step 1~4 UI**가 모두 end-to-end로 연동됩니다.

## Slide 18 — 배포 & 운영 구성 (1분)

배포 구성입니다.

**Client**는 Vercel에 배포했습니다. Next.js SSR과 Edge Runtime을 활용합니다.

CI/CD는 GitHub Actions + CodeRabbit AI 자동 리뷰, feature → dev → main PR 프로세스를 사용했습니다. 개발에 Claude Code + MCP 서버를 활용했고, RAG는 팀원 3명이 각 1개씩 전담 구축했습니다.

## Slide 19 — 핵심 개발 이슈 & 해결 (1) (1분 30초)

핵심 이슈입니다. 먼저 RAG 데이터와 평가 관련입니다.

**R3 법령 RAG 데이터 수집·청킹** — 항 단위 청킹에서 같은 조 내 항들이 서로 경쟁하는 문제와, 한국어 BM25가 "원격의료"를 분해 못 하는 문제가 있었습니다. 3단계 실험으로 조 단위(C8) + E1 + SPLADE를 확정, 최종 MH-Recall 96.7% 달성.

**R2 승인 사례 데이터 정제** — 281건 중 정보 부족 50건을 GPT-4o로 보강해 MRR 0.91→0.95. Score threshold는 점수 분포가 좁아서 정답까지 필터링되는 걸 확인하고 사용하지 않기로 했습니다.

**RAG 공통 교훈** — R2와 R3 모두에서, **데이터 정제와 청킹 전략이 임베딩 모델 교체보다 훨씬 큰 성능 향상**을 가져왔습니다. 비싼 모델로 바꾸는 것보다 데이터를 잘 다듬는 게 더 효과적이었습니다. "한 번에 하나의 변수만 변경" 원칙으로 재현 가능한 실험을 수행했습니다.

## Slide 20 — 핵심 개발 이슈 & 해결 (2) (1분 15초)

기술 구현 이슈 네 가지입니다.

**HWP 비표준 포맷 파싱** — olefile + zlib + UTF-16LE + 정규식으로 파서를 직접 구현해서 11종 양식을 안정적으로 파싱합니다. HWP 바이너리 구조에 대한 깊은 이해와 역공학 경험을 쌓았습니다.

**Agent 간 데이터 일관성** — Canonical Structure 설계로 4개 Agent가 동일한 JSON을 공유, 불일치를 근본 방지했습니다. 새 Agent 추가 시에도 확장 가능합니다.

**SSE 실시간 진행 표시** — LangGraph astream + progress_store + FastAPI SSE로 "지금 어떤 단계 처리 중"인지 실시간 표시. 체감 대기시간을 크게 줄였습니다.

**ChromaDB → Qdrant 전환** — BaseVectorStore 추상 클래스로 교체 가능하게 설계해서, R3만 먼저 Qdrant로 전환할 수 있었습니다.
