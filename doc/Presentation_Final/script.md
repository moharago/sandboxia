## Slide 3 — 기획 배경 & 목적 (2분)

먼저 기획 배경입니다. 저희가 해결하고자 한 문제는 크게 세 가지입니다.

**첫째, 신청서 작성이 매우 복잡합니다.** 규제 샌드박스에는 신속확인, 임시허가, 실증특례라는 세 가지 트랙이 있는데요, 오른쪽 그림에 보시는 것처럼 트랙마다 양식도 다르고, 필수 항목도 다르고, 법적 논점도 전혀 다릅니다. 컨설턴트 입장에서 보면, 먼저 기업이 어떤 기술을 쓰는지 이해해야 하고, 그 기술이 어떤 규제에 걸리는지 법령을 찾아야 하고, "왜 이 규제를 풀어야 하는가"에 대한 논리까지 구성해야 합니다. 이 세 가지를 동시에 해야 하는 상당히 복합적인 작업이에요.

**둘째, 유사 사례 조사가 비효율적입니다.** 컨설턴트가 신청서를 쓸 때 가장 먼저 하는 일이 "과거에 비슷한 서비스가 승인받은 적이 있나?" 확인하는 건데요, 이걸 하려면 과거 승인 사례 PDF를 하나하나 열어서 수작업으로 검색해야 합니다. 어떤 법령이 쟁점이었는지, 어떤 논리로 심사를 통과했는지를 일일이 파악하는 데 시간이 많이 걸리고, 정보가 여기저기 흩어져 있어서 파편화 문제도 있습니다.

**셋째, 수요는 늘어나는데 리소스는 제한적입니다.** 2026년 산업융합촉진법 개정으로 샌드박스 수요가 계속 증가하고 있습니다. 컨설턴트 한 명이 처리해야 할 건수는 늘어나는데, 조사하고 작성하는 데 걸리는 시간은 줄어들지 않습니다. 결국 사람을 더 뽑거나, 도구로 효율을 높이거나 둘 중 하나인데, 저희는 후자를 선택한 겁니다.

그래서 저희의 **타겟 사용자**는 규제 샌드박스 신청을 도와주는 컨설턴트입니다. 구체적으로는 ICT융합 지원기관 소속 공공기관 컨설턴트와 민간 행정사, 이 두 그룹이 주요 사용자입니다.

## Slide 4 — 프로젝트 목표 & 핵심 가치 (1분 30초)

프로젝트 목표입니다.

정량적으로는 세 가지 달성 지표를 잡았습니다. **4단계 AI 파이프라인을 실제 동작하는 수준으로 완성**하는 것, **과거 승인 사례 281건을 구조화된 DB로 구축**하는 것, 그리고 **3개 도메인별 RAG 검색 시스템을 구축**하는 것입니다.

정성적으로는 세 가지를 목표로 했습니다. 컨설턴트의 조사와 초안 작성 시간을 획기적으로 단축하고, 대상성 판단의 초기 품질을 안정화하며, 그동안 개인 경험에 의존하던 노하우를 데이터 기반으로 보조하는 겁니다. 경험이 적은 컨설턴트도 일정 수준의 품질을 낼 수 있게 하자는 취지입니다.

오른쪽에 핵심 기능 세 가지를 정리했는데요.

**하나**, 4단계 AI 파이프라인입니다. 서비스 구조화 → 대상성 판단 → 트랙 추천 → 신청서 초안, 이 네 단계를 전담 에이전트가 순차적으로 수행합니다. 실제 컨설턴트의 업무 흐름을 그대로 AI 파이프라인으로 옮긴 겁니다.

**둘**, 3-RAG 기반 도메인 검색입니다. 승인 사례 281건, 11개 법률, 제도 절차 문서를 의미 기반으로 검색해서 유사 사례 매칭과 관련 법령 자동 연결을 해줍니다. 키워드가 정확히 일치하지 않아도 의미가 비슷하면 찾아주는 거죠.

**셋**, 신청서 초안 자동 생성입니다. 트랙별 양식에 맞춰 DOCX 파일로 초안을 만들어주고, 리치 에디터에서 편집도 가능합니다. 여기서 중요한 설계 원칙 하나 — **AI는 보조 역할이고, 최종 판단은 반드시 컨설턴트가 한다**는 것입니다. AI가 판단을 대신 내리는 게 아니라, 근거와 초안을 제공해서 컨설턴트가 더 빠르고 정확하게 일할 수 있도록 돕는 겁니다.

## Slide 5 — 기술 스택 (1분)

기술 스택입니다. 간단히 짚어드리겠습니다.

**프론트엔드**는 Next.js 16과 React 19를 기반으로 합니다. TailwindCSS 4로 스타일링하고, TanStack Query 5로 서버 상태를 관리합니다. 신청서 초안을 편집하려면 단순 텍스트 입력이 아니라 서식이 있는 에디터가 필요했기 때문에 Tiptap 3 리치 에디터를 도입했습니다. 폼 검증에는 React Hook Form과 Zod를 사용했고, UI 컴포넌트는 Radix UI 기반입니다.

**백엔드**는 FastAPI 위에 LangGraph 1.0으로 멀티 에이전트를 구현했습니다. LLM은 GPT-4o, 임베딩은 text-embedding-3-large를 3개 RAG 공통으로 사용합니다. 벡터 DB로는 Qdrant를 사용했고, RAG 평가에는 RAGAS 프레임워크를 활용했습니다.

**인프라**는 Supabase로 PostgreSQL 데이터베이스와 Google OAuth 인증, 파일 스토리지를 통합 관리합니다. Docker Compose로 서버와 Qdrant를 컨테이너화했고, CI/CD는 GitHub Actions, 코드 리뷰는 CodeRabbit AI가 PR마다 자동으로 수행합니다.

## Slide 6 — 컨설팅 워크플로우 × AI 파이프라인 (2분)

이제 핵심인 서비스 기능 설명입니다. 이 슬라이드가 전체 서비스를 한눈에 보여주는 가장 중요한 슬라이드입니다.

왼쪽이 **4단계 컨설팅 워크플로우**인데요, 실제 컨설턴트가 업무를 처리하는 순서를 그대로 AI 파이프라인으로 설계했습니다. "AI를 위한 워크플로우"가 아니라, "컨설턴트의 업무 흐름에 AI를 끼워 넣은 것"이 핵심입니다.

**Step 1**은 기업 서비스 접수와 구조화입니다. 컨설턴트가 기업의 HWP 신청서를 업로드하면, Service Structurer Agent가 비정형 문서를 파싱해서 통일된 구조로 변환합니다. 이 구조가 이후 모든 단계의 입력이 됩니다.

**Step 2**는 샌드박스 대상성 검토입니다. "이 서비스가 규제 샌드박스에 신청할 수 있는 대상이 맞는가?"를 판단합니다. Eligibility Evaluator Agent가 3개 RAG를 모두 병렬로 검색해서 근거를 수집한 뒤, LLM이 종합 판정합니다. 컨설턴트가 AI 판정을 Override할 수 있도록 설계했습니다.

**Step 3**은 적합 트랙 선정입니다. 세 가지 트랙 중 어디에 신청하는 게 가장 유리한지, Track Recommender Agent가 체크리스트 기반 스코어링과 유사 사례 매칭으로 추천합니다.

**Step 4**는 신청서 초안 작성입니다. Application Drafter Agent가 추천된 트랙의 양식에 맞춰 자동으로 초안을 생성하고, 컨설턴트가 에디터에서 편집한 뒤 DOCX로 다운로드합니다.

오른쪽에 **Agent × RAG 활용 매트릭스**가 있는데요, 이걸 보시면 Step 1만 RAG를 사용하지 않고, Step 2, 3, 4는 R1(제도/절차), R2(승인사례), R3(법령) 세 가지 RAG를 모두 활용하는 것을 볼 수 있습니다. 이 3개 RAG는 에이전트들이 공유하는 인프라로, 팀원 3명이 각각 하나씩 담당해서 구축했습니다. 아래쪽에 각 RAG의 데이터 소스와 규모가 정리되어 있습니다.

## Slide 7 — Step 1: 서비스 구조화 (1분 30초)

Step 1을 자세히 보겠습니다.

이 단계가 왜 중요하냐면, 기업이 보내는 HWP 신청서는 양식도 제각각이고, 같은 내용이 여러 장에 흩어져 있기도 하고, 어떤 건 손으로 쓴 것도 있습니다. 이 비정형 데이터를 이후 모든 에이전트가 공통으로 사용할 수 있는 **Canonical Structure**, 즉 통일된 JSON 구조로 변환하는 게 Step 1의 목표입니다. 이 JSON이 제대로 만들어져야 뒤의 판단, 추천, 초안 생성이 정확해집니다.

2-Node 워크플로우로 동작합니다.

**첫 번째 노드, Parse HWP**에서는 HWP 바이너리를 파싱합니다. HWP는 한글 워드프로세서의 비공개 바이너리 포맷이라 일반적인 방법으로는 텍스트를 꺼낼 수 없어요. 그래서 olefile이라는 라이브러리로 OLE 컨테이너 구조를 열고, 안에 있는 데이터를 zlib 압축 해제하고, UTF-16LE로 디코딩합니다. 그런 다음 정규식 패턴으로 회사명, 서비스명, 체크박스 상태, 서명 여부 등 50개 이상의 필드를 자동 추출합니다.

**두 번째 노드, Build Structure**에서는 최대 4장의 HWP를 8개 필드 그룹(회사, 서비스, 기술, 규제, 사업, 안전, 사유, 신청)으로 분류하고, first-win 전략으로 병합합니다. 같은 필드가 여러 문서에 나오면 먼저 나온 값을 우선 적용한다는 뜻이에요. 그 다음 GPT-4o(temperature 0.1)가 이 병합 데이터를 Canonical JSON으로 변환합니다.

여기서 중요한 설계 원칙이 있습니다. **체크박스 상태, 서명 여부, 날짜 같은 사실 데이터는 HWP 파서에서 직접 추출한 값을 LLM 출력보다 항상 우선 적용**합니다. LLM이 "이 칸은 체크되어 있을 것 같다"고 추측하는 걸 막은 거죠. AI가 판단할 영역과 그대로 복사할 영역을 명확히 분리한 겁니다.

## Slide 8 — Step 2: 대상성 판단 (1분 45초)

Step 2는 "이 서비스가 규제 샌드박스에 신청할 자격이 있는가?"를 판단하는 단계입니다. 이게 컨설턴트 입장에서 첫 번째 관문이에요. 자격이 안 되는 걸 준비하면 시간 낭비니까요. 4개 노드로 구성됩니다.

**Screen** 노드에서 먼저 키워드 스크리닝을 합니다. 5개 규제 도메인(healthcare, finance, mobility, data, telecom)별로 10~15개의 키워드 사전을 만들어놨는데, 서비스 설명을 스캔해서 "원격의료", "자율주행", "암호화폐" 같은 고위험 키워드를 탐지합니다. 왜 LLM 호출 전에 키워드 스크리닝을 먼저 하냐면, 이 서비스가 어떤 규제 도메인과 관련이 있는지를 빠르게 파악해서, 다음 단계의 RAG 검색 방향을 잡아주기 위해서입니다. 불필요한 도메인까지 전부 검색하면 비용과 시간이 낭비되니까요.

**Search All RAG** 노드에서는 3개 RAG 소스를 **병렬로 검색**합니다. ThreadPoolExecutor를 사용해서 R1에서 제도/절차/요건을, R2에서 유사 승인 사례 top-5를, R3에서 도메인별 법령을 동시에 가져옵니다. 순차적으로 검색하면 시간이 3배 걸리니까, 병렬 처리로 응답 속도를 맞춘 겁니다.

**Compose Decision** 노드에서 GPT-4o가 키워드 스크리닝 결과와 RAG 3종 검색 결과를 통합해서 종합 판정을 내립니다. 판정 결과는 "필요성 높음", "병행 검토", "대상 아님" 세 가지입니다. 여기서 신뢰도 점수는 LLM이 "난 80% 확신해"라고 직접 매기는 게 아니라, 수집된 근거의 수량과 일관성에 기반한 규칙으로 별도 계산합니다. LLM의 자기 확신은 신뢰하기 어렵기 때문입니다.

마지막 **Generate Evidence** 노드에서는 9개의 LLM 호출을 ThreadPoolExecutor로 병렬 실행해서, 각 RAG 항목에 대해 2~3문장의 근거 설명을 생성합니다. 이게 UI의 좌측 판정 요약과 우측 승인 사례/규제 법령 탭에 표시됩니다.

그리고 핵심 설계 원칙 — **AI는 보조 역할이고 최종 판단은 컨설턴트가 직접 선택**합니다. AI가 "대상 아님"이라고 판정해도 컨설턴트가 "아니야, 이건 대상이야"라고 Override할 수 있는 UI를 만들었습니다. 도메인 전문가의 판단이 항상 우선입니다.

## Slide 9 — Step 3: 트랙 추천 (1분 45초)

Step 3은 세 가지 트랙 중 가장 적합한 트랙을 추천하는 단계입니다.

세 트랙이 어떻게 다른지 쉽게 설명드리면요. **실증특례**는 "아직 안전성이 충분히 검증되지 않았으니, 제한된 지역이나 기간 안에서 테스트해볼게요"라는 트랙입니다. **임시허가**는 "안전성은 이미 검증됐는데, 현행법에 허가 근거 자체가 없으니 임시로 허가해주세요"라는 트랙입니다. **신속확인**은 "우리 서비스에 기존 규제가 적용되는 건지 안 되는 건지 불명확하니, 빨리 확인만 해주세요"라는 트랙입니다. 상황이 다르면 들어갈 트랙도 달라지는 거죠.

이 추천을 4개 노드로 자동화합니다.

첫째, **Retrieve Cases**에서 R2 승인 사례를 의미 기반으로 검색합니다. 전체 사례에서 한 번에 검색한 뒤 트랙별로 자동 그룹핑합니다. 처음에는 트랙별로 쪼개서 검색했는데, 신속확인 데이터가 0건이라 결과가 안 나오는 버그가 있었어요. 전체 검색 후 그룹핑으로 바꾸니까 해결됐습니다.

둘째, **Score All Tracks**에서 트랙별 **체크리스트 기반 LLM 스코어링**을 합니다. 각 트랙마다 4~5개 평가 기준이 있고, LLM이 각 기준을 true/false + 사유로 평가합니다. 예를 들어 실증특례면 "실증 테스트가 필요한가?", "안전성이 아직 미검증인가?", "제한 범위 설정이 가능한가?" 등을 봅니다. 점수는 (충족 수/전체) × 100으로 계산합니다. 그리고 R3 법령 RAG로 **차단 규제**를 탐지하면 **최대 -30점 페널티**를 부과합니다. 아무리 체크리스트 점수가 높아도 법적으로 명확히 막혀있으면 해당 트랙 순위를 낮춰야 하니까요.

셋째, **Retrieve Definitions**에서 R1을 조회해 3개 트랙의 공식 정의와 심사 기준을 가져옵니다.

넷째, **Generate Recommendation**에서 R1(정의) + R2(사례) + R3(법령) + 스코어를 종합해서 트랙별 5개 추천 사유와 근거를 생성합니다. LLM이 생성한 인용이 실제 RAG 결과와 일치하는지 출처 검증 후처리도 수행합니다. "이 사례를 참고했다"고 하면서 실제로는 없는 사례를 인용하는 환각을 방지하기 위해서입니다.

## Slide 10 — Step 4: 신청서 초안 생성 (1분 45초)

마지막 Step 4, 신청서 초안 생성입니다. 컨설턴트에게 가장 직접적인 가치를 제공하는 단계이기도 합니다.

추천된 트랙에 맞는 양식을 자동으로 채워서 초안을 만듭니다. 트랙별로 양식이 다른데요, 신속확인은 비교적 간단해서 2장짜리 양식을 쓰고, 임시허가와 실증특례는 사업 개요, 특례 범위, 안전 방안 등 상세한 내용이 필요해서 각 4장짜리 양식을 사용합니다.

3-Node 워크플로우인데, 핵심은 세 번째 노드의 **5단계 필드 채우기**입니다. 왜 5단계로 나눴냐면, "AI가 모든 칸을 다 쓴다"가 아니라 "AI가 써야 할 칸과 그대로 복사할 칸을 명확히 구분한다"는 설계 철학 때문입니다.

**3-1 Pass-through**: 회사명, 대표자, 매출액, 인력 수 같은 사실 데이터는 LLM을 거치지 않고 Canonical Structure에서 그대로 복사합니다. AI가 숫자를 바꾸거나 이름을 틀리게 쓸 위험을 원천 차단한 거죠.
**3-2 Section Text**: HWP에서 추출한 서비스 설명, 사업 계획 같은 원문은 미리 만들어둔 매핑 테이블에 따라 해당 섹션에 직접 배치합니다.
**3-3 트랙 전환 처리**: 기업이 올린 HWP가 실증특례용인데 추천 트랙이 임시허가인 경우가 있습니다. 이때 양식이 다르니까, 선택된 트랙 양식에 맞게 사유 필드를 재생성합니다.
**3-4 LLM 생성**: 위 3단계를 거치고도 비어있는 서술형 필드만 GPT-4o(temperature 0.3)가 작성합니다. R1에서 작성 가이드를, R2에서 유사 사례를 참고해서 씁니다.
**3-5 트랙별 사유**: 실증특례나 임시허가 양식에 있는 체크박스 사유와 근거를 자동 생성합니다.

아래에 **3종 데이터 분리 설계**가 있는데, 이것도 중요합니다. 원본을 보존하는 application_input, AI 분석에 사용하는 canonical, 컨설턴트가 편집하는 application_draft를 완전히 분리해서 관리합니다. 컨설턴트가 초안을 아무리 수정해도 원본 데이터와 AI 분석 결과는 항상 보존됩니다.

생성된 초안은 **Tiptap 리치 에디터**에서 서식을 유지하면서 편집할 수 있고, 최종적으로 **docxtpl** 템플릿 엔진을 통해 DOCX 파일로 렌더링해서 다운로드할 수 있습니다.

## Slide 11 — 기능 구현 기술 아키텍처 (1분 15초)

전체 시스템 아키텍처를 한눈에 보여주는 슬라이드입니다.

왼쪽의 **Client**는 Next.js 16 기반으로 포트 3000에서 동작합니다. TanStack Query 5로 서버 데이터를 캐싱하고 동기화하며, Zustand로 클라이언트 로컬 상태를 관리합니다. Tiptap 3은 신청서 초안 편집에 사용됩니다.

가운데 **Server**는 FastAPI 위에 LangGraph 1.0을 올렸고, Agent 4개와 RAG 3개가 동작합니다. 클라이언트와는 일반 요청은 REST API로, 분석 진행 상황은 SSE(Server-Sent Events) 스트림으로 실시간 전달합니다. SSE를 쓴 이유는, Agent 분석이 최대 2분까지 걸리는데 그동안 "지금 어떤 단계를 처리 중입니다"를 보여줘야 하기 때문입니다.

LLM은 생성에 GPT-4o를 사용하고, 임베딩은 text-embedding-3-large를 3개 RAG 공통으로 사용합니다. R3 법령 RAG 평가에서 고차원 모델이 10%p 더 높은 성능을 보였고, 운영 환경에서 임베딩 모델을 RAG별로 분리 관리하기 어려워 성능이 더 좋은 E1으로 통일했습니다.

오른쪽 **Data** 레이어는 Supabase Cloud에서 PostgreSQL, Google OAuth 인증, 파일 Storage를 통합 관리하고, Qdrant가 벡터 스토어로 포트 6333에서 Docker Compose 안에서 돌아갑니다.

## Slide 12 — RAG 툴 구성 (1분 30초)

이제 RAG 시스템을 자세히 보겠습니다. 저희 시스템의 핵심 검색 인프라이자, 에이전트 성능을 좌우하는 가장 중요한 컴포넌트입니다.

**R1, 규제제도 & 절차 RAG**입니다. ICT 규제 샌드박스의 제도 문서를 기반으로 합니다. 10개 카테고리(트랙 정의, 절차, 요건 등) × 4개 트랙 × 6개 부처 구조로 되어 있어서, track + category + ministry 필터로 정밀 검색이 가능합니다. "실증특례의 과기부 심사 기준이 뭐야?"처럼 구체적인 질의에 답할 수 있습니다.

**R2, 승인 사례 RAG**입니다. 실제 승인/반려 사례 281건을 담고 있는데요, 특이한 점은 1건이 1문서, 청킹을 하지 않았다는 겁니다. 사례 데이터가 최대 1,211자, 평균 1,000자 미만으로, 임베딩 모델 토큰 한도의 10%도 안 되기 때문에 쪼갤 이유가 없었습니다. 그리고 281건 중 정보가 부족한 50건을 GPT-4o로 보강(enriched)해서 검색 품질을 높였습니다. 어떻게 보강했는지는 다음 슬라이드에서 설명드리겠습니다.

**R3, 도메인별 법령 RAG**입니다. 국가법령정보센터 Open API에서 6개 도메인의 11개 법률을 수집하고, 조(article) 단위로 청킹해서 1,188개 청크를 만들었습니다. 평균 청크 크기는 480자입니다. 차단 규제 탐지에 핵심적으로 사용되는데, "이 서비스가 어떤 법 몇 조에 의해 막히는가?"를 찾아주는 역할입니다.

공통 인프라로는 Qdrant 벡터 DB를 사용하고, **BaseVectorStore**라는 추상 클래스를 만들어서 persistent(파일 저장), http(서버 연결), ephemeral(평가용 임시) 세 가지 모드로 동작할 수 있게 설계했습니다. 평가할 때는 ephemeral 모드로 임시 컬렉션을 만들어서 운영 DB를 오염시키지 않습니다.

## Slide 13 — R1 RAG 평가 (2분)

이제 RAG 성능 평가 파트입니다. 저희 프로젝트에서 가장 실험적이고 공을 들인 부분입니다. R1부터 보겠습니다.

R1은 147개 청크, 30개 질의-정답 쌍으로 3단계 최적화를 수행했습니다. 평가 지표는 MH-Recall(정답 청크 중 몇 개를 찾았는가), MRR(정답이 검색 결과 상위에 있는가), Latency(응답 속도)를 사용했습니다.

**1단계, 임베딩 모델 비교**입니다. 3종을 비교했는데, E0 text-embedding-3-small(1536차원), E1 text-embedding-3-large(3072차원), E2 Solar Embedding(4096차원)입니다. 결과적으로 **E1이 MH-Recall 73.3%, MRR 88.3%로 성능과 속도 모두 1위**였습니다. E2 Solar는 차원이 4096으로 가장 컸지만 성능은 E1보다 낮고 지연시간은 4배 길었습니다. 규제 전문 용어처럼 의미가 복잡하고 밀도가 높은 텍스트에는 고차원 임베딩이 효과적이되, 무조건 높다고 좋은 건 아니라는 걸 확인했습니다.

**2단계, 검색 전략 비교**입니다. 벡터 검색, 하이브리드 검색(50/50, 80/20), 쿼리 리라이팅, 리랭킹을 비교했는데요. 의외의 결과가 나왔습니다. **하이브리드 검색은 오히려 MRR이 21%p나 떨어졌습니다.** R1 데이터가 구조화된 제도 문서라서 키워드 매칭(BM25)이 오히려 노이즈를 만든 겁니다. 리랭킹은 MH-Recall이 90%로 크게 올라갔지만 지연 시간이 19.6초로, 기본 151ms의 **129배**가 되어서 실시간 서비스에는 부적합했습니다. 쿼리 리라이팅은 효과가 거의 없었고요. 결국 **벡터 검색 단독이 가장 효율적**이었습니다.

**3단계, 생성 품질 검증**입니다. Faithfulness, 즉 검색 결과에 근거해서 환각 없이 답변하는 비율이 **98.67%**로 매우 높았습니다. 또한 검색 쿼리에 트랙명을 프리픽스로 붙이면 MH-Recall이 73.3%에서 88.3%로 15%p 올라가는 것도 발견했습니다.

최종 확정: **E1 + Vector Search + gpt-4o-mini**입니다.

## Slide 14 — R2 RAG 평가 (2분 15초)

R2 승인 사례 RAG는 4단계 최적화를 수행했습니다. R2는 "비슷한 서비스가 과거에 승인받은 사례를 찾아주는" RAG이기 때문에, 실제 검색 품질이 곧 서비스 품질과 직결됩니다.

**1단계, 데이터 전략 비교**입니다. 같은 281건 데이터를 세 가지 방식으로 가공해서 비교했습니다. structured는 정형 필드(서비스명, 도메인, 트랙 등)만 사용, enriched는 GPT-4o로 원문에서 핵심 키워드와 기술 설명을 추출해서 보강한 버전, hybrid는 원문 전체와 정형 필드를 합친 버전입니다. 결과적으로 **enriched가 Recall 0.858로 기본 대비 7.3%p 높았습니다.** 왜 그랬냐면, 원본 사례 중에 서비스 설명이 너무 짧거나 모호한 게 있었거든요. GPT-4o가 "이 사례의 핵심 기술은 A이고, 규제 쟁점은 B"라고 명시적으로 뽑아줬더니 임베딩 벡터가 더 정확해진 겁니다.

**2단계, 임베딩 모델 비교**입니다. API 모델 2종(E0 3-small, E1 3-large)과 로컬 모델 2종(BGE-M3, KURE-v1)을 비교했는데, 모델 간 차이가 약 2.5%p로 매우 소폭이었습니다. R1, R3와 달리 왜 차이가 작았냐면, R2는 1건이 1문서이고 청킹이 없어서, 문서 자체가 하나의 완결된 의미 단위이기 때문입니다. 평가 결과만 보면 E0가 MRR 0.950으로 1위였지만, 운영 환경에서 임베딩 모델을 RAG별로 따로 관리하기 어려워 R1/R3과 동일하게 **E1 text-embedding-3-large로 통일**했습니다. 어차피 성능 차이가 2.5%p 이내여서 통일해도 손해가 거의 없었습니다.

**3단계, 생성 모델 비교**입니다. RAGAS 프레임워크로 4개 모델을 비교했는데, **gpt-4o가 Faithfulness 0.975, Answer Relevancy 0.718, 지연시간 P50 2.9초로 세 지표 모두 1위**였습니다. 흥미로운 건 gpt-4o-mini가 지연시간 9.0초로 가장 느렸다는 점입니다. mini 모델이 오히려 더 오래 걸린 건 의외였습니다.

**4단계, 구성 튜닝**입니다. 가장 실용적인 단계였는데, top_k를 3, 5, 7, 10으로 바꿔봤습니다. k=3은 MH-Recall이 0.717로 정답을 놓치는 경우가 많았고, k=7은 MH-Recall은 0.933으로 올라갔지만 Answer Relevancy가 8.4%p 떨어졌습니다. 관련 없는 문서가 같이 들어오면서 LLM이 혼란을 느낀 거죠. **k=5가 검색 품질과 생성 품질의 최적 균형점**이었습니다.

Score threshold 실험도 했는데, 유사도 점수 분포가 min 0.08에서 max 0.58로 너무 좁아서, threshold 0.3을 적용하면 전체의 52%가 필터링되고 정답 문서까지 잘려나갔습니다. 결론은 **threshold 불필요**, 사용하지 않는 것이 최선이었습니다.

최종 확정: **enriched + E1 + gpt-4o + top_k=5 + threshold 없음**입니다.

## Slide 15 — R3 RAG 평가 (2분 15초)

R3 법령 RAG는 3단계 최적화를 수행했습니다. R3는 "이 서비스가 어떤 법에 막히는가?"를 찾아주는 RAG이라서, 정확도가 특히 중요합니다. 잘못된 법령을 인용하면 신청서 전체의 신뢰도가 무너지니까요.

**1단계, 청킹 전략 비교**입니다. 법령 조문을 어떤 단위로 쪼갤지 9개 조합을 실험했습니다. 처음에는 항(paragraph) 단위로 쪼갰는데, 같은 조에 속하는 1항, 2항, 3항이 서로 경쟁하면서 정작 필요한 조문이 검색 결과에서 밀려나는 현상이 있었습니다. 예를 들어 "제10조"에 대해 물었는데, 10조 1항과 10조 3항이 각각 별도 청크로 올라와서 서로 점수를 깎아먹은 거죠. **조 단위로 합치니까 이 경쟁이 사라지면서 MH-Recall이 13%p 올라갔습니다.** 최종적으로 C8, 즉 조 단위 + 법률명 프리픽스("정보통신융합법 제36조") + 짧은 조항은 별도 청크로 분리하는 전략을 채택했습니다.

**2단계, 임베딩 모델 비교**입니다. API 3종, 로컬 3종 총 6개 모델을 비교했습니다. **E1 text-embedding-3-large가 MH-Recall 96.7%로 E0(86.7%) 대비 10%p 높았습니다.** R1과 마찬가지로, 법률 텍스트는 한 문장에 여러 개념이 압축되어 있는 "의미 밀도가 높은" 텍스트여서, 저차원 벡터로는 의미를 충분히 담지 못하고 고차원 임베딩이 효과적이었습니다. 반면 E2 Solar는 차원이 4096이었지만 지연시간이 939ms로 5배 느리면서 성능은 E1과 동일했습니다.

**3단계, 벡터 DB와 하이브리드 검색 비교**입니다. 여기서 가장 흥미로운 발견이 있었습니다. 기존 ChromaDB Dense 검색도 MH-Recall 93.3%로 나쁘지 않았지만, Qdrant에 **SPLADE 하이브리드 검색**을 도입하니까 96.7%로 올라갔습니다. 그런데 같은 Qdrant인데 **BM25**와 조합하면 오히려 88.3%로 떨어졌습니다.

왜 이런 차이가 났냐면, 한국어 BM25는 띄어쓰기 기반으로 토큰을 나눕니다. "원격의료"를 하나의 토큰으로 취급해서, "원격"이나 "의료"로는 매칭이 안 돼요. 반면 SPLADE는 WordPiece 토크나이저를 사용합니다. "원격의료"를 "원격", "##의료"로 서브워드 분해하기 때문에 부분 매칭이 가능합니다. 이 차이가 **MRR 8~13%p** 차이를 만든 겁니다. 한국어 RAG에서 하이브리드 검색을 할 때 BM25가 아닌 SPLADE가 훨씬 효과적이라는 중요한 교훈을 얻었습니다.

최종 확정: **C8 + E1 + Qdrant SPLADE**, MH-Recall 96.7%, Recall 91.7%입니다.

## Slide 16 — Agent 아키텍처 (1분 30초)

전체 Agent 아키텍처를 한눈에 보여드리겠습니다.

네 개 Agent가 순차적으로 동작하는 파이프라인인데, 각 Agent는 LangGraph의 StateGraph로 구현되어 있고, 독립적인 노드와 엣지로 구성됩니다.

**Agent 1 Service Structurer**가 HWP를 파싱하고 구조화해서 **Canonical Structure**를 만듭니다. 이 Canonical Structure가 전체 파이프라인의 핵심 데이터 계약입니다. 이후 Agent 2, 3, 4가 모두 이 동일한 JSON을 입력으로 받기 때문에, 에이전트마다 데이터를 다르게 해석하는 불일치 문제를 근본적으로 방지할 수 있었습니다. 새로운 Agent를 추가하더라도 이 Canonical Structure만 참조하면 됩니다.

**Agent 2 Eligibility Evaluator**는 키워드 스크리닝으로 방향을 잡은 뒤 R1, R2, R3를 ThreadPoolExecutor로 병렬 검색해서 대상성을 판정합니다. 근거 설명 생성도 9개 LLM 호출을 병렬로 처리해서 속도를 높였습니다.

**Agent 3 Track Recommender**는 유사 사례를 검색하고, 트랙별 체크리스트 스코어링을 수행한 뒤, R3 차단 규제 탐지로 최대 -30점 페널티를 반영해서 최종 추천 순위를 결정합니다.

**Agent 4 Application Drafter**는 양식 스키마를 로드하고, RAG에서 작성 가이드와 유사 사례를 검색한 뒤, Pass-through(복사)와 LLM 생성을 5단계로 조합해서 초안을 만듭니다. Tiptap 에디터에서 편집 가능한 형태로 출력하고, docxtpl로 DOCX 렌더링합니다.

하단의 End-to-End Pipeline Flow를 보시면, HWP 업로드 → Agent 1 구조화 → Canonical Structure 생성 → Agent 2/3/4 분석 → 최종 결과(판정 + 추천 + 초안)까지의 전체 데이터 흐름을 한눈에 확인하실 수 있습니다.

## Slide 17 — 서비스 완성 (시연) (1분 30초)

실제 서비스를 시연으로 보여드리겠습니다.

시연 흐름은 다섯 단계입니다. 먼저 Google OAuth로 로그인해서 대시보드에 진입합니다. 대시보드에서는 진행 중인 프로젝트 목록과 각 프로젝트의 Step 1~4 진행 상태를 한눈에 볼 수 있습니다.

## Slide 18 — 배포 & 운영 구성 (1분 15초)

배포 구성입니다.

**Client**는 Vercel에 배포했습니다. Next.js의 SSR과 Edge Runtime을 활용하고, 빌드 시 자동으로 최적화됩니다.

**Server**는 Docker Compose로 FastAPI 서버와 Qdrant 벡터 DB를 하나로 묶어서 배포합니다. Docker 이미지는 ghcr.io, GitHub Container Registry에 저장합니다. 컨테이너화한 이유는 로컬 개발 환경과 운영 환경의 차이를 없애고, Qdrant 같은 외부 서비스도 함께 버전 관리하기 위해서입니다.

**Database**는 Supabase Cloud를 사용합니다. PostgreSQL에 RLS(Row Level Security)를 적용해서 사용자별 데이터 접근을 행 단위로 제어합니다. 컨설턴트 A가 맡은 프로젝트를 컨설턴트 B가 볼 수 없도록 하는 거죠.

**CI/CD 파이프라인**은 dev 브랜치에 Push하면 GitHub Actions가 빌드와 테스트를 돌리고, PR이 올라오면 CodeRabbit이 AI 코드 리뷰를 자동으로 수행합니다. main에 머지되면 자동 배포됩니다.

오른쪽에 협업 구성이 있는데, feature 브랜치에서 dev, dev에서 main으로의 PR + 코드 리뷰 프로세스를 사용했습니다. 개발 도구로는 Claude Code와 MCP 서버를 활용했고, RAG는 팀원 3명이 R1, R2, R3를 각각 하나씩 전담해서 구축했습니다.

## Slide 19 — 핵심 개발 이슈 & 해결 (1) (1분 45초)

개발 과정에서 겪었던 핵심 이슈들입니다. 먼저 RAG 데이터와 평가 관련 이슈 두 가지입니다.

**첫 번째, R3 법령 RAG의 데이터 수집과 청킹입니다.** 처음에 법령 조문을 항(paragraph) 단위로 쪼갰는데, 두 가지 문제가 생겼습니다. 하나는 항이 너무 짧아서 의미가 불완전하다는 것, 다른 하나는 같은 조에 속하는 여러 항이 검색 결과에서 서로 경쟁하면서 정작 필요한 조문이 밀려나는 것이었습니다. 거기에 더해 한국어 BM25는 띄어쓰기 기반 토큰화라 "원격의료"를 "원격"과 "의료"로 나누지 못하는 문제도 있었고요.

해결책으로 국가법령정보센터 API에서 11개 법률을 수집한 뒤 체계적으로 3단계 실험을 했습니다. 청킹 전략 9개를 비교해서 조 단위(C8)를 확정하니 MH-Recall이 13%p 올랐고, 임베딩 6종 비교로 E1을 선택하니 10%p 더 올랐고, Qdrant + SPLADE 하이브리드로 한국어 서브워드 문제를 해결하면서 최종 MH-Recall 96.7%를 달성했습니다. 단계별로 하나의 변수만 바꿔가며 실험한 덕분에, 각 선택의 근거가 명확합니다.

**두 번째, R2 승인 사례 데이터 정제입니다.** 281건의 승인 사례 중 서비스 설명이 충실하게 기록된 건이 제한적이었고, ICT 도메인에 편중되어 바이오나 모빌리티 쪽은 데이터가 부족했습니다. GPT-4o로 50건을 보강(enriched)하고, 30개 테스트 쿼리로 클러스터 분석을 해서 gold_cases를 전수 조사했습니다. enriched 적용 후 MRR이 0.91에서 0.95로 올라갔습니다. Score threshold(0.3) 실험도 했는데, 유사도 점수 분포가 너무 좁아서 전체의 52%가 필터링되며 정답까지 잘려나가는 걸 확인하고 사용하지 않기로 했습니다.

아래에 **RAG 공통 교훈**을 적어뒀습니다. R2와 R3 모두에서 확인된 건, **데이터 정제와 청킹 전략이 임베딩 모델 교체보다 훨씬 큰 성능 향상**을 가져왔다는 것입니다. 비싼 모델로 바꾸는 것보다, 데이터를 잘 다듬는 게 더 효과적이었습니다. 그리고 "한 번에 하나의 변수만 변경"하는 원칙을 지켜서, 모든 실험을 재현 가능하게 만들었습니다.

## Slide 20 — 핵심 개발 이슈 & 해결 (2) (1분 45초)

기술 구현 과정에서 겪은 이슈 네 가지입니다.

**HWP 비표준 포맷 파싱**. 가장 처음 부딪힌 벽입니다. HWP는 마이크로소프트의 OLE 컨테이너 위에 독자 바이너리를 얹은 비공개 포맷이라, Python에 안정적인 파서가 없습니다. olefile로 OLE 구조를 열고, zlib 압축을 해제하고, UTF-16LE로 디코딩하고, 정규식 패턴 매칭으로 50개 이상의 필드를 추출하는 파서를 직접 구현했습니다. 결과적으로 11종 신청서 양식을 모두 안정적으로 파싱할 수 있게 됐습니다. HWP 바이너리 구조에 대한 깊은 이해와 역공학 경험을 쌓은 소중한 과정이었습니다.

**Agent 간 데이터 일관성**. 4개 Agent가 각각 입력 데이터를 다르게 해석하면 문제가 됩니다. 예를 들어 Agent 2는 "의료기기"로 판단했는데 Agent 3은 "헬스케어"로 인식하면 판정, 추천, 초안 사이에 불일치가 생기죠. 이걸 해결하기 위해 **Canonical Structure**를 설계했습니다. Agent 1이 만든 통일 JSON을 나머지 Agent 3개가 동일하게 공유하도록 했습니다. 새 Agent를 추가할 때도 이 구조만 참조하면 되니까 확장성도 확보했습니다.

**실시간 진행 상태 표시(SSE)**. Agent 분석이 30초에서 2분까지 걸리는데, 그동안 사용자에게 빈 화면만 보여주면 "멈춘 건가?" 하고 이탈하게 됩니다. 이걸 해결하기 위해 LangGraph의 astream API로 노드 진입/완료 이벤트를 스트리밍하고, progress_store에 단계별 상태를 저장한 뒤, FastAPI SSE 엔드포인트로 클라이언트에 실시간 전달합니다. 사용자는 "키워드 스크리닝 중...", "RAG 검색 중...", "판정 생성 중..." 같은 메시지를 프로그레스 바와 함께 볼 수 있습니다. 체감 대기시간이 크게 줄었습니다.

**ChromaDB에서 Qdrant로 전환**. R3 법령 RAG에서 Dense 검색만으로는 법령 조문의 키워드 매칭이 부족했습니다. Qdrant + SPLADE 하이브리드를 도입했는데, 이때 기존 R1/R2의 ChromaDB를 갑자기 바꾸면 안 되니까, **BaseVectorStore라는 추상 클래스**를 먼저 설계했습니다. 이 인터페이스를 통해 ChromaDB와 Qdrant를 교체 가능하게 만들어서, R3만 먼저 Qdrant로 전환하고 나머지는 나중에 마이그레이션할 수 있었습니다.

## Slide 21 — 프로젝트 회고 & 향후 과제 (1분 45초)

마지막으로 회고입니다.

먼저 **중간 발표 피드백 반영** 내용입니다. 세 가지 피드백을 받았고 모두 반영했습니다. "입력 문서에서 부족한 부분을 판단하는 기능이 필요하다"는 피드백에는 Agent 1에 Uncertainty Detector를 구현해서 미비 항목을 자동 탐지하도록 했습니다. "비용, 시간, 퀄리티 기준으로 RAG 성능을 정량 평가해야 한다"는 피드백에는 R2에서 6단계 평가 파이프라인을 수립해서, 데이터 전략부터 구성 튜닝까지 체계적으로 비교했습니다. "유저 개인화 페이지와 대시보드가 필요하다"는 피드백에는 프로젝트별 대시보드와 Step 1~4 진행 상태 관리 UI를 구현했습니다.

**프로젝트를 통해 얻은 것** 세 가지입니다. 첫째, LangGraph로 4개 Agent를 실제 서비스에 적용하면서 **멀티 에이전트 아키텍처를 실무 수준에서 경험**했습니다. Canonical Structure 설계, 에이전트 간 데이터 흐름, 병렬 처리 등 이론과 다른 실전의 문제들을 해결했습니다. 둘째, 규제 샌드박스라는 **특수 도메인의 전문성을 학습**했습니다. 3개 트랙의 요건과 절차 차이를 이해하고, AI 보조 도구의 UX를 설계하는 경험을 쌓았습니다. 셋째, CodeRabbit AI 코드 리뷰 + Claude Code + MCP 서버 등 **AI 도구를 활용한 협업 방식**을 경험했습니다.

**현재 한계**로는 세 가지가 있습니다. 실제 컨설턴트 피드백 없이 내부 테스트만 수행했다는 점, 281건 사례에서 ICT에 편중되고 바이오/모빌리티 데이터가 부족한 점, Uncertainty Detector가 누락 항목은 탐지하지만 항목별 "충분히 작성됐는가"의 기준이 아직 부재하다는 점입니다.

**향후 과제**로는 전략 자문 Agent(유사 사례 클러스터링 → 승인 패턴 추출)와 QA 리스크 체크 Agent(심사자 관점 시나리오 생성) 추가, 실서비스 배포 후 컨설턴트 피드백 수집 → RAG와 프롬프트 개선 루프 구축, 승인 사례 데이터 자동 수집 파이프라인 구축을 계획하고 있습니다.
