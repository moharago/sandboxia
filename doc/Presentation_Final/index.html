01 — Purpose

기획 배경 & 목적

01

신청서 작성의 복잡성

3개 트랙(신속확인/임시허가/실증특례)마다 양식, 필수 항목, 법적 논점이 모두 다름.
컨설턴트는 기업 기술 이해 + 관련 규제 법령 탐색 + 논리 구성을 동시에 수행

02

유사 사례 조사의 비효율

과거 승인 사례 PDF를 수작업으로 검색/분석. 쟁점 법령, 논리 전개 방식을 일일이 파악해야 하며 정보가 파편화

03

수요 급증 vs 리소스 한계

2026년 산업융합촉진법 개정으로 샌드박스 수요 증가 전망. 컨설턴트 1인당 처리량은 늘지만 조사/작성 시간은 고정

Target User

규제 샌드박스
신청 컨설턴트

공공기관 컨설턴트 (ICT융합 지원기관)

04 — Step 3

Step 3: 트랙 추천 Agent 3

4-Node 워크플로우

1. Retrieve Cases — R2 유사 승인 사례 검색
의미 기반 검색 → 트랙별 자동 그룹핑 (전체 사례에서 한 번에)
↓
2. Score All Tracks — 3개 트랙 체크리스트 스코어링 (0~100)
트랙별 평가 기준(4~5개)에 따라 LLM이 적합도 점수 산출
↓
3. Retrieve Definitions — R1 트랙 정의 + R3 도메인 법령 조회
차단 규제 탐지 → 해당 트랙 최대 -30점 페널티
↓
4. Generate Recommendation — 최종 순위 결정 + 추천 사유 생성
스코어 + 사례 + 법령을 종합하여 트랙별 추천 근거 작성

06 — RAG System

RAG 툴 구성 — 3-RAG 공유 아키텍처

모든 에이전트가 공용으로 사용하는 핵심 검색 인프라. 팀원 3명이 각 1개씩 담당 구축.

R1. 규제제도 & 절차

데이터: ICT 규제샌드박스 제도 문서

구조: 10개 카테고리 x 4개 트랙 x 6개 부처

필터: track + category + ministry

기능: 트랙 정의, 제출요건, 심사기준, 트랙 비교

사용: Agent 2, 3, 4

R2. 승인 사례

데이터: 승인/반려 사례 281건

구조: 1건 = 1문서 (청킹 없음)

보강: GPT-4o로 50건 enriched

기능: 유사사례 검색, 승인 패턴 분석, 중복 제거

사용: Agent 2, 3

R3. 도메인별 법령

데이터: 국가법령정보센터 API 11개 법률

구조: 조(article) 단위 청킹, 6개 도메인

청크: 1,188개 (평균 480자)

기능: 도메인별 법령 검색, 차단 규제 탐지

사용: Agent 1, 2, 3

공통 인프라

ChromaDB (Vector DB) text-embedding-3-small BaseVectorStore ABC persistent / http / ephemeral 3모드

10 — Retrospective

프로젝트 회고

기술적 측면

  • LangGraph 멀티 에이전트 패턴을 실서비스에 적용하며 에이전트 간 데이터 흐름 설계의 중요성 체감
  • RAG 시스템에서 데이터 품질(수집/정제/임베딩)이 결과 품질에 미치는 영향이 절대적
  • 체계적 평가(3단계) 없이는 RAG 성능 개선이 '감'에 의존하게 됨
  • Next.js 16 + TanStack Query + Zustand 조합으로 서버/클라이언트 상태 깔끔 분리

협업 측면

  • feature → dev → main 브랜치 전략과 PR 워크플로우로 코드 충돌 최소화
  • CodeRabbit AI 코드 리뷰로 품질 관리 자동화
  • Claude Code + MCP 서버 활용으로 개발 생산성 향상
  • RAG 팀원별 1개씩 분업으로 전문성 집중

도메인 측면

  • 규제 샌드박스 제도의 복잡성(3개 트랙, 각 트랙별 요건/절차 차이) 학습에 상당한 시간 소요
  • 승인 사례 데이터 수집/정제가 예상보다 어려움
  • AI가 "법적 결론"이 아닌 "참고용 보조"임을 명확히 하는 UX 설계의 중요성
