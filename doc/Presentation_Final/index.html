04 — Step 1

Step 1: 서비스 구조화 Agent 1

기업이 업로드한 HWP 신청서(1~4장)와 컨설턴트 메모를 분석하여, 이후 모든 에이전트가 공통으로 사용하는 Canonical Structure (통일 JSON)를 생성합니다.

2-Node 워크플로우

1. Parse HWP — HWP 바이너리 파싱 + 필드 추출
olefile로 OLE 구조 읽기 → PrvText(UTF-16LE) / BodyText(zlib 해제) 중 긴 텍스트 채택
정규식으로 회사명, 서비스명, 체크박스 상태(✓/□) 등 50+개 필드 자동 추출
↓
2. Build Structure — 문서 병합 + LLM 구조화
문서 병합: 최대 4장의 HWP를 8개 필드그룹(회사/서비스/기술/규제/사업/안전/사유/신청)으로 분류, first-win 전략으로 통합
LLM 구조화: GPT-4o(temp 0.1)가 병합 데이터를 Canonical JSON으로 변환. 체크박스/서명/날짜는 HWP 파서 값을 우선 적용

04 — Step 2

Step 2: 대상성 판단 Agent 2

서비스가 규제 샌드박스 신청 대상인지 판단합니다. 키워드 스크리닝 → RAG 검색 → LLM 종합 판단 → 근거 생성의 4단계 파이프라인으로 구성됩니다.

4-Node 워크플로우

1. Screen — 규제 도메인 키워드 사전 매칭
5개 도메인(healthcare, finance, mobility, data, telecom) × 10~15개 키워드로 서비스 설명 스캔
→ 고위험 키워드 탐지(원격의료, 자율주행, 암호화폐 등) → risk_signals + search_keywords 도출
↓
2. Search All RAG — 3개 RAG 소스 병렬 검색 (ThreadPoolExecutor)
R1 제도/절차/요건   R2 유사 승인사례 (서비스 설명 기반 의미 검색, top-5)   R3 도메인별 법령 (도메인당 top-3)
↓
3. Compose Decision — GPT-4o 종합 판단
키워드 스크리닝 결과 + RAG 3종 결과를 통합하여 eligibility_label 결정
→ 신뢰도는 근거 수량 기반 규칙으로 별도 계산 (LLM 의존 X)
↓
4. Generate Evidence — RAG 결과 → UI용 근거 데이터 변환
ThreadPoolExecutor(9 workers)로 RAG 항목별 2~3문장 설명 병렬 생성
→ judgment_summary(좌측 패널) + approval_cases(우측 탭) + regulations(우측 탭)

04 — Step 3

Step 3: 트랙 추천 Agent 3

3개 트랙(실증특례/임시허가/신속확인) 중 가장 적합한 트랙을 추천합니다. 유사 사례 검색 → 체크리스트 스코어링 → 트랙 정의 조회 → 추천 사유 생성의 4단계로 동작합니다.

4-Node 워크플로우

1. Retrieve Cases — R2 유사 승인 사례 의미 검색
서비스 설명을 임베딩하여 전체 사례에서 한 번에 검색(top-3) → 트랙별 자동 그룹핑
→ 이후 스코어링/추천 사유 생성에서 참고 근거로 활용
↓
2. Score All Tracks — 트랙별 체크리스트 기반 LLM 스코어링 (0~100)
트랙마다 4~5개 평가 기준을 LLM이 true/false + 사유로 평가 → fit_score = (충족 수/전체) × 100
R3 차단 규제 탐지 시 최대 -30점 페널티 적용 후 순위 결정
↓
3. Retrieve Definitions — R1 트랙 정의/요건 조회
3개 트랙의 공식 정의, 신청 절차, 심사 기준을 RAG에서 검색 (트랙당 top-3)
→ 다음 노드에서 추천 사유 작성 시 근거 자료로 활용
↓
4. Generate Recommendation — 최종 순위 + 추천 사유 생성
R1(정의) + R2(사례) + R3(법령) + 스코어를 종합하여 트랙별 5개 추천 사유 + 근거 작성
→ 출처 검증: LLM이 생성한 인용이 실제 RAG 결과와 일치하는지 후처리로 검증

04 — Step 4

Step 4: 신청서 초안 생성 Agent 4

추천된 트랙에 맞는 신청서 양식을 자동으로 채워 초안을 생성합니다. 사실 데이터는 그대로 복사하고, 서술형 필드만 LLM이 작성합니다.

3-Node 워크플로우

1. Load Form Schema — 트랙별 양식 JSON 로드
추천 트랙에 맞는 폼 스키마 파일 선택 (신속확인 2장, 임시허가 4장, 실증특례 4장)
↓
2. Retrieve Context — R1 + R2 + R3 RAG 검색
R1 신청 요건/작성 가이드 + 심사 기준   R2 유사 승인 사례 (top-5)   R3 도메인 법령
↓
3. Generate Draft — 5단계 필드 채우기
3-1. Pass-through: 회사/재무/인력 등 사실 데이터 → 그대로 복사 (LLM 미사용)
3-2. Section Text: HWP 원문(서비스 설명, 사업 계획 등) → 매핑 테이블로 직접 배치
3-3. 트랙 전환 처리: 다른 트랙 HWP → 선택 트랙 양식으로 사유 필드 재생성
3-4. LLM 생성: 비어있는 서술형 필드만 GPT-4o(temp 0.3)로 작성
3-5. 트랙별 사유: 실증/임시허가 체크박스 사유 + 근거 자동 생성

06 — RAG System

RAG 툴 구성 — 3-RAG 공유 아키텍처

모든 에이전트가 공용으로 사용하는 핵심 검색 인프라. 팀원 3명이 각 1개씩 담당 구축.

R1. 규제제도 & 절차

데이터: ICT 규제샌드박스 제도 문서

구조: 10개 카테고리 x 4개 트랙 x 6개 부처

필터: track + category + ministry

기능: 트랙 정의, 제출요건, 심사기준, 트랙 비교

사용: Agent 2, 3, 4

R2. 승인 사례

데이터: 승인/반려 사례 281건

구조: 1건 = 1문서 (청킹 없음)

보강: GPT-4o로 50건 enriched

기능: 유사사례 검색, 승인 패턴 분석, 중복 제거

사용: Agent 2, 3, 4

10 — Dev Issues (1/2)

핵심 개발 이슈 & 해결 (1)

R3 법령 RAG — 데이터 수집부터 청킹 전략까지

Problem

법령 조문은 항(paragraph) 단위가 너무 짧고, 같은 조의 항들이 서로 경쟁하여 검색 누락 발생. 한국어 BM25는 띄어쓰기 기반 토큰화로 서브워드 분해 불가.

Solution

국가법령정보센터 API에서 11개 법률 수집 후 3단계 실험:

1
청킹 9개 전략(C0~C8) → 조 단위(C8) 확정
2
임베딩 6종 → E1(3-large) MH-R +10%p
3
Hybrid Qdrant+SPLADE MRR +8~13%p
Result

조 단위가 항 단위 대비 MH-R +13%p. SPLADE의 WordPiece 토크나이저가 한국어 서브워드 일관 분해. 최종 MH-R 96.7%.

R2 규제 도메인 데이터 수집 & 정제

Problem

승인 사례 281건 중 구조화 가능 데이터가 제한적. 법령 데이터는 조문 구조 불일관. 도메인 편중(ICT 과다, 바이오 부족).

Solution

GPT-4o로 50건 보강(enriched) + 30개 쿼리 클러스터 분석으로 gold_cases 전수 조사. 법령은 조 단위(C8) 청킹 + 메타데이터 보존.

Result

Enriched로 MRR 0.91→0.95. Score threshold(0.3)는 52% 필터링으로 유해 → 사용하지 않음.

RAG 공통 교훈

R2와 R3 모두 데이터 정제와 청킹 전략이 임베딩 모델 교체보다 큰 성능 향상을 가져왔다. “한 번에 하나의 변수만 변경” 원칙으로 재현 가능한 최적 구성 도출.

10 — Dev Issues (2/2)

핵심 개발 이슈 & 해결 (2)

HWP 비표준 포맷 파싱

Problem

HWP는 비공개 바이너리 포맷이며, 기업마다 양식/구조가 상이. Python에 안정적인 HWP 파서가 없어 표준 라이브러리로는 텍스트 추출 불가.

Solution

olefile로 OLE 컨테이너 파싱 → zlib 압축 해제 → UTF-16LE 디코딩 → 정규식 패턴 매칭으로 50+ 필드 추출. 8개 필드 그룹 병합(first-win 전략)으로 최대 4개 HWP 통합.

Result

11종 신청서 양식을 안정적으로 파싱. HWP 바이너리 구조에 대한 깊은 이해와 역공학 경험 축적.

Agent 간 데이터 일관성

Problem

4개 Agent가 각각 입력 데이터를 다르게 해석하면 판정/추천/초안 간 불일치 발생. Agent 2가 “의료기기”로 판단했는데 Agent 3이 “헬스케어”로 인식하는 등의 문제.

Solution

Canonical Structure(통일 JSON 스키마) 설계. Agent 1이 생성한 구조화 데이터를 나머지 3개 Agent가 동일한 입력으로 공유. 서비스명, 도메인, 핵심기술 등 필드를 표준화.

Result

Agent 간 데이터 불일치 해소. 새 Agent 추가 시에도 Canonical Structure만 참조하면 되어 확장성 확보.

실시간 진행 상태 표시 (SSE)

Problem

Agent 분석이 30초~2분 소요되는데, 사용자에게 진행 상태를 전혀 보여줄 수 없었음. 응답 전까지 빈 화면만 표시되어 UX가 매우 불량.

Solution

LangGraph astream으로 노드별 이벤트 스트리밍 → progress_store에 단계별 상태 저장 → FastAPI SSE 엔드포인트로 클라이언트에 실시간 전달. 프론트엔드에서 단계별 프로그레스 바 표시.

Result

“어떤 Agent가 지금 무엇을 하고 있는지” 실시간 표시. 체감 대기시간 크게 감소.

ChromaDB → Qdrant 하이브리드 검색 전환

Problem

ChromaDB Dense 검색만으로는 법령 조문의 키워드 매칭이 부족. 한국어 BM25는 띄어쓰기 기반 토큰화로 서브워드 분해 불가.

Solution

Qdrant + SPLADE Hybrid Search 도입. BaseVectorStore ABC로 추상화하여 ChromaDB/Qdrant 교체 가능한 설계. Docker Compose로 Qdrant :6333 통합.

Result

SPLADE가 BM25 대비 MRR +8~13%p. R3 최종 MH-R 96.7%, Recall 91.7% 달성.
